
# طرح جامع ماژول مدیریت لوحه و اعزام پرسنل
## Roster & Shift Module v2.6

## ۱) هدف ماژول

ماژول لوحه باید فایل‌های عملیاتی روزانه مثل Excel و PDF را که معمولاً به‌صورت جدولی، چندستونه، اسکن‌شده یا نیمه‌ساختاریافته تولید می‌شوند، به یک سیستم دیجیتال تبدیل کند که بتواند:

- برنامه حرکت قطارها را استخراج کند.
- راهبران H1 و H2 را تشخیص دهد.
- جهت حرکت قطار را مشخص کند.
- ساعت رسیدن و حرکت را ساختاریافته ذخیره کند.
- لوحه روزانه را برای هر راهبر شخصی‌سازی کند.
- تغییرات نسخه‌های مختلف لوحه را ردیابی کند.
- تأیید رؤیت و آمادگی راهبر را ثبت کند.
- در شرایط قطع LTE و تونل به‌صورت آفلاین کار کند.
- داده‌های استخراج‌شده را به تقویم کاری، حضور و غیاب، چک‌لیست و داشبورد مدیریتی وصل کند.

# ۲) شناخت ساختار واقعی فایل لوحه

بر اساس تصویر و PDF، لوحه چند ویژگی مهم دارد:

## ۲.۱ مشخصات بالای لوحه

در سربرگ لوحه اطلاعاتی مثل موارد زیر دیده می‌شود:

- عنوان: گزارش لوحه اعزام
- تاریخ لوحه
- روز هفته
- خط: خط ۱
- شماره پردازش
- نوع زمان‌بندی / عنوان برنامه
- تاریخ اخذ گزارش
- نام کاربر صادرکننده
- مسیرهای دوطرفه:
    - از تجریش تا شهرری
    - از شهرری تا تجریش

این اطلاعات باید به‌عنوان **Metadata** ذخیره شوند، نه اینکه فقط در فایل باقی بمانند.

## ۲.۲ ساختار دوطرفه جدول

لوحه در هر صفحه معمولاً دو بخش موازی دارد:

- سمت راست: یک جهت حرکت
- سمت چپ: جهت مقابل

هر بخش ستون‌هایی شبیه این دارد:

- شماره ردیف / نوبت
- شماره قطار
- ستون‌های عملیاتی مثل `H1`
- ستون `H2`
- ستون `T`
- ستون `R`
- زمان حرکت
- زمان رسیدن

## ۲.۳ تکرار ساختار در چند صفحه

فایل PDF شما ۳ صفحه دارد و ردیف‌ها به‌صورت پیوسته ادامه پیدا می‌کنند:

- صفحه ۱: ردیف‌های ابتدایی
- صفحه ۲: ادامه ردیف‌ها
- صفحه ۳: ردیف‌های پایانی

پس سیستم باید بتواند:

- صفحات متعدد را یکی کند.
- شماره ردیف‌ها را ادامه‌دار تشخیص دهد.
- از تکرار سربرگ‌ها جلوگیری کند.
- داده‌های سمت چپ و راست هر صفحه را جداگانه استخراج کند.
- خروجی نهایی را در یک جدول واحد ذخیره کند.

# ۳) مدل داده پیشنهادی حرفه‌ای‌تر

مدلی که نوشتی خوب است، اما بهتر است بین **لوحه، نسخه لوحه، سفر، تخصیص پرسنل و رخداد عملیاتی** تفکیک ایجاد شود.

## ۳.۱ RosterDay — لوحه روزانه

```tsx
RosterDay {
  id: string
  lineCode: string              // LINE_1
  jalaliDate: string            // 1404/07/16
  gregorianDate: Date
  weekday: string               // چهارشنبه
  title: string                 // گزارش لوحه اعزام
  schedulingTitle: string       // روز عادی / کاهش شانت / ...
  processingNumber: number      // مثلا 7
  reportTakenAt: DateTime
  issuedByName: string
  sourceType: 'EXCEL' | 'PDF' | 'IMAGE'
  status: 'DRAFT' | 'PROCESSING' | 'REVIEW' | 'PUBLISHED' | 'ARCHIVED'
  createdAt: DateTime
  updatedAt: DateTime
}
```

## ۳.۲ RosterVersion — نسخه لوحه

چون لوحه ممکن است در طول روز اصلاح شود، نسخه را از خود لوحه جدا کن.

```tsx
RosterVersion {
  id: string
  rosterDayId: string
  versionNo: number             // 1, 2, 3
  fileUrl: string
  originalFileName: string
  uploadedById: string
  uploadedAt: DateTime
  publishedById?: string
  publishedAt?: DateTime
  changeReason?: string
  changeSummary?: string
  status: 'UPLOADED' | 'PARSED' | 'VALIDATED' | 'PUBLISHED' | 'REJECTED'
  checksum: string              // برای جلوگیری از آپلود تکراری
}
```

## ۳.۳ Trip — حرکت / سفر عملیاتی

هر ردیف واقعی لوحه باید به یک Trip تبدیل شود.

```tsx
Trip {
  id: string
  rosterVersionId: string
  rowNo: number                 // شماره ردیف جدول
  trainNumber?: string          // شماره قطار
  direction: 'TAJRISH_TO_SHAHRREY' | 'SHAHRREY_TO_TAJRISH'
  originStationCode: string     // TAJRISH یا SHAHRREY
  destinationStationCode: string
  departureTime: string         // HH:mm:ss
  arrivalTime: string           // HH:mm:ss
  h1RawName?: string
  h2RawName?: string
  tRawValue?: string
  rRawValue?: string
  operationalNote?: string
  status: 'NORMAL' | 'EMPTY' | 'CANCELLED' | 'DEPOT' | 'SPECIAL'
}
```

## ۳.۴ TripAssignment — تخصیص پرسنل به سفر

بهتر است H1 و H2 را مستقیماً داخل Trip نهایی نکنیم؛ چون ممکن است در آینده نقش‌های بیشتری اضافه شود.

```tsx
TripAssignment {
  id: string
  tripId: string
  role: 'H1' | 'H2' | 'T' | 'R' | 'RESERVE' | 'SUPERVISOR'
  rawName: string
  matchedUserId?: string
  personnelNo?: string
  matchScore?: number
  matchStatus: 'AUTO_MATCHED' | 'NEEDS_REVIEW' | 'MANUAL_MATCHED' | 'UNMATCHED'
  confirmedById?: string
  confirmedAt?: DateTime
}
```

### مزیت این مدل

اگر در آینده ستون‌های T و R هم معنی عملیاتی مشخص‌تری پیدا کنند، بدون تغییر مدل اصلی می‌توان آن‌ها را به نقش‌های جدید تبدیل کرد.

# ۴) موتور استخراج لوحه از Excel و PDF

## ۴.۱ مسیر پیشنهادی برای Excel

برای فایل `loheadi.xlsx`:

1. خواندن workbook
2. تشخیص sheet فعال
3. تشخیص محدوده جدول
4. تشخیص دو بلوک راست و چپ
5. تشخیص ستون‌های هر بلوک
6. استخراج ردیف‌ها
7. نرمال‌سازی زمان‌ها
8. نرمال‌سازی نام‌ها
9. تشخیص جهت حرکت
10. تولید Trip و TripAssignment

## ۴.۲ مسیر پیشنهادی برای PDF

برای `111.pdf` چون خروجی متنی خام دقیق نیست و فقط مقدارهایی مثل `H1 H2 T R` خوب آمده، باید از ترکیب دو روش استفاده شود:

### روش اول: OCR جدولی

- تبدیل هر صفحه PDF به تصویر
- تشخیص خطوط جدول
- تشخیص سلول‌ها
- OCR هر سلول
- بازسازی جدول

### روش دوم: Template-based Extraction

چون ظاهر لوحه ثابت است، می‌توان مختصات جدول را تعریف کرد:

- محدوده جدول سمت راست
- محدوده جدول سمت چپ
- ستون شماره قطار
- ستون H1
- ستون H2
- ستون T
- ستون R
- ستون زمان حرکت
- ستون زمان رسیدن

این روش برای فایل‌های ثابت سازمانی بسیار دقیق‌تر از OCR عمومی است.

## ۴.۳ پیشنهاد حرفه‌ای

برای PDF این فایل‌ها، از **Template Parser** استفاده کن، نه فقط OCR ساده.

یعنی سیستم یک قالب ذخیره کند:

```tsx
RosterTemplate {
  id: string
  name: string
  sourceType: 'PDF' | 'EXCEL'
  pageWidth: number
  pageHeight: number
  rightBlock: Box
  leftBlock: Box
  columns: ColumnMapping[]
  headerZones: HeaderMapping[]
}
```

# ۵) Visual Mapper برای پنل ادمین

این یکی از مهم‌ترین بخش‌هاست.

## ۵.۱ چرا لازم است؟

چون فایل‌های لوحه ممکن است در آینده کمی تغییر کنند:

- جابه‌جایی ستون‌ها
- اضافه شدن ستون جدید
- تغییر فونت
- تغییر عنوان‌ها
- تغییر تعداد صفحات
- خروجی گرفتن از سیستم دیگر

پس نباید همه‌چیز hard-code شود.

## ۵.۲ امکانات Visual Mapper

ادمین بتواند روی پیش‌نمایش فایل:

- ناحیه جدول راست را انتخاب کند.
- ناحیه جدول چپ را انتخاب کند.
- ستون‌های H1/H2/T/R را مشخص کند.
- ستون زمان حرکت و رسیدن را مشخص کند.
- جهت حرکت هر بخش را تعیین کند.
- الگو را ذخیره کند.
- برای فایل‌های بعدی همان الگو را استفاده کند.

## ۵.۳ خروجی Mapper

```tsx
ColumnMapping {
  block: 'RIGHT' | 'LEFT'
  sourceColumnIndex: number
  targetField:
    | 'rowNo'
    | 'trainNumber'
    | 'H1'
    | 'H2'
    | 'T'
    | 'R'
    | 'departureTime'
    | 'arrivalTime'
}
```

# ۶) پاک‌سازی و نرمال‌سازی داده‌ها

در لوحه‌های فارسی، خطاهای زیادی ممکن است رخ دهد:

- چسبیدن نام و نام خانوادگی
- اشتباه OCR بین «ی» و «ي»
- اشتباه OCR بین «ک» و «ك»
- حذف فاصله
- نیم‌فاصله‌های ناهماهنگ
- اشتباه در اعداد فارسی و انگلیسی
- خواندن اشتباه ساعت
- خالی بودن H2 یا T/R

## ۶.۱ Normalizer فارسی

قبل از matching، همه متن‌ها باید نرمال شوند:

```
ي → ی
ك → ک
ة → ه
ۀ → ه
حذف کشیدگی ـ
تبدیل اعداد عربی/فارسی به لاتین برای پردازش
حذف فاصله‌های اضافه
یکسان‌سازی نیم‌فاصله
```

## ۶.۲ Fuzzy Matching

برای تطبیق نام‌ها با کاربران:

- Jaro-Winkler برای نام‌های کوتاه
- Levenshtein برای خطاهای تایپی
- Token Sort Ratio برای جابه‌جایی نام و نام خانوادگی
- Soundex فارسی یا الگوریتم مشابه در نسخه پیشرفته

## ۶.۳ آستانه پیشنهادی

| امتیاز تطبیق | وضعیت |
| --- | --- |
| ۹۵ تا ۱۰۰ | تطبیق خودکار قطعی |
| ۸۵ تا ۹۵ | تطبیق خودکار با هشدار |
| ۷۰ تا ۸۵ | نیازمند بررسی ادمین |
| کمتر از ۷۰ | بدون تطبیق |

## ۶.۴ Staging Area

قبل از انتشار، همه داده‌ها باید وارد محیط بررسی شوند:

- سبز: معتبر
- زرد: نیازمند بررسی
- قرمز: خطای جدی
- خاکستری: ردیف خالی یا غیرعملیاتی

# ۷) اعتبارسنجی عملیاتی و ایمنی

## ۷.۱ اعتبارسنجی ساختاری

- تاریخ لوحه معتبر است.
- شماره پردازش استخراج شده.
- هر Trip دارای زمان حرکت و رسیدن است.
- جهت حرکت مشخص است.
- شماره ردیف تکراری نیست.
- زمان‌ها صعودی هستند.
- صفحه‌ها درست ترکیب شده‌اند.

## ۷.۲ اعتبارسنجی پرسنلی

- H1 در بانک پرسنلی وجود دارد.
- H2 اگر ثبت شده، قابل تطبیق است.
- یک نفر در دو قطار هم‌زمان ثبت نشده.
- فرد مرخصی ندارد.
- فرد در وضعیت فعال است.
- گواهی لازم معتبر است.

## ۷.۳ اعتبارسنجی Anti-Fatigue

برای هر راهبر:

- مجموع ساعت کاری روزانه
- مجموع زمان رانندگی
- تعداد سفرها
- فاصله بین رسیدن و حرکت بعدی
- تعداد شانت‌ها
- تعداد سفرهای پشت‌سرهم
- پایان آخرین سفر و شروع شیفت بعدی

## ۷.۴ سطح هشدار

```tsx
ValidationIssue {
  severity: 'INFO' | 'WARNING' | 'ERROR' | 'CRITICAL'
  type: string
  message: string
  affectedTripId?: string
  affectedUserId?: string
  suggestedAction?: string
}
```

# ۹) تجربه کاربری موبایل راهبر

## ۹.۱ صفحه «لوحه امروز»

در موبایل راهبر باید خلاصه‌ای واضح نمایش داده شود:

```
لوحه امروز — چهارشنبه ۱۴۰۴/۰۷/۱۶
خط ۱ — نسخه ۲
وضعیت: نیازمند تأیید رؤیت

سفر بعدی شما:
قطار ۵۰۹
نقش: H1
حرکت: ۰۵:۳۰
رسیدن: ۰۶:۳۵
جهت: تجریش ← شهرری

[تأیید رؤیت] [اعلام آمادگی] [گزارش مغایرت]
```

## ۹.۲ Timeline View

مناسب برای دیدن کل روز:

- سفرهای انجام‌شده
- سفر جاری
- سفر بعدی
- شانت بین دو سفر
- زمان استراحت
- تغییرات اخیر
- هشدارها

## ۹.۳ Card View

مناسب برای محیط کابین:

- فونت درشت
- کنتراست بالا
- دکمه‌های بزرگ
- اطلاعات کم اما ضروری
- مناسب حالت تاریک
- بدون شلوغی جدول اصلی

## ۹.۴ حالت «کابین»

یک حالت ویژه پیشنهاد می‌کنم:

### Cabin Mode

وقتی راهبر در زمان نزدیک به حرکت است:

- فقط اطلاعات سفر فعلی نمایش داده شود.
- نوتیفیکیشن‌های غیرضروری مخفی شوند.
- دکمه SOS در دسترس باشد.
- دکمه اعلام آمادگی فعال باشد.
- صفحه با کنتراست بالا و فونت بزرگ باشد.
- مصرف باتری کاهش یابد.

# ۱۱) Offline-First و LTE

## ۱۱.۱ داده‌هایی که باید آفلاین ذخیره شوند

- لوحه امروز
- لوحه فردا در صورت انتشار
- سفرهای شخصی کاربر
- نسخه فعلی لوحه
- وضعیت تأییدهای ثبت‌نشده
- اطلاعات ضروری همکار کابین
- شماره تماس سرشیفت
- آخرین پیام‌های بحرانی

## ۱۱.۲ صف عملیات آفلاین

اقدامات زیر در صورت قطع LTE باید صف شوند:

- تأیید رؤیت
- اعلام آمادگی
- تحویل کابین
- گزارش مغایرت
- گزارش تأخیر
- ثبت چک‌لیست
- ثبت حضور

## ۱۱.۳ سیاست تعارض

اگر کاربر آفلاین یک عملیات ثبت کند و هم‌زمان نسخه لوحه تغییر کند:

- عملیات کاربر حفظ شود.
- نسخه جدید لوحه دریافت شود.
- سیستم بررسی کند عملیات مربوط به سفر حذف‌شده یا تغییرکرده بوده یا نه.
- در صورت تعارض، به سرشیفت هشدار داده شود.
- به کاربر پیام واضح نمایش داده شود.

# ۱۶) نکات اصلاحی روی متن فعلی تو

چند پیشنهاد دقیق برای بهترشدن متن فعلی:

## ۱۶.۱ از `Trip.h1_driver_id` و `Trip.h2_driver_id` فاصله بگیر

بهتر است از `TripAssignment` استفاده کنی؛ چون آینده‌پذیرتر است.

## ۱۶.۲ فیلد `date: Date` کافی نیست

چون تاریخ شمسی در فایل مبناست، بهتر است هر دو نگهداری شود:

- `jalaliDate`
- `gregorianDate`

## ۱۶.۳ `direction` باید بر اساس بلوک جدول تعیین شود

در PDF، هر نیمه جدول نماینده یک جهت است. پس Parser باید بداند:

- بلوک راست = مثلاً شهرری به تجریش
- بلوک چپ = مثلاً تجریش به شهرری

البته این نگاشت باید در Template قابل تنظیم باشد.

## ۱۶.۴ `operationalStatus` را از اول خیلی محدود نکن

بهتر است statusهای بیشتری داشته باشی:

```
NORMAL
CANCELLED
DEPOT_DISPATCH
DEPOT_RETURN
RESERVE
SPECIAL
EMPTY
MANUAL_REVIEW
```

## ۱۶.۵ حتماً `rawValue` ذخیره کن

هر داده استخراج‌شده باید مقدار خام هم داشته باشد. چون اگر OCR یا Excel اشتباه خواند، بتوان بررسی کرد.

# ۱۷) جمع‌بندی نسخه نهایی پیشنهادی

ماژول لوحه باید به‌صورت یک خط پردازش کامل طراحی شود:

```
فایل لوحه
→ تشخیص قالب
→ استخراج Metadata
→ استخراج دو بلوک جهت حرکت
→ استخراج ردیف‌ها
→ نرمال‌سازی فارسی
→ تطبیق نام با پرسنل
→ ساخت Trip و Assignment
→ اعتبارسنجی ایمنی
→ پیش‌نمایش و اصلاح دستی
→ انتشار نسخه
→ اعلان به راهبران
→ ذخیره آفلاین
→ تأیید رؤیت و اعلام آمادگی
→ اتصال به حضور، تقویم، چک‌لیست و داشبورد
```

طرح  که لوحه را فقط «یک جدول برنامه حرکت» نبینی؛ بلکه آن را به **هسته عملیاتی اعزام روزانه** تبدیل کنی. این ماژول می‌تواند قلب سیستم باشد، چون از روی آن تقویم، حضور، آماده‌باش، هشدار خستگی، گزارش عملکرد، چک‌لیست و حتی AI Assistant تغذیه می‌شوند.

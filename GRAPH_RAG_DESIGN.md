# 🧠 طرح نهایی: موتور دانش گرافی دستیار AI (Hybrid GraphRAG)

**نسخه:** 2.0 — نهایی، یکپارچهشده با `AI_ASSISTANT_DESIGN.md`
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**جایگاه:** این سند، **لایه دانش و بازیابی** دستیار AI را بازطراحی میکند. معماری پنجلایه (L0–L4)، پرسوناها، ابزارها و پنل دستیار سر جای خود میمانند؛ گراف دانش بهعنوان **ستون فقرات جدید بازیابی** زیر آنها مینشیند و یک خط پاسخ جدید (L3-G) اضافه میکند.

---

## ۰. اصلاح انتظارات (تعهدات قابل دفاع)

| ادعای وسوسهانگیز | فرمولبندی مهندسی این طرح |
| :--- | :--- |
| «دقت ۱۰۰٪ بدون توهم» | **کاهش چشمگیر توهم + ارجاع اجباری هر پاسخ به سند/صفحه (Grounding)** + پاسخ صادقانه «در اسناد یافت نشد» وقتی مسیری در گراف نیست |
| «مصرف توکن صفر» | **انتقال هزینه از "هر پرسش" به "هر ورود سند"** (یکبار، محلی، رایگان) — کاهش ۷۰–۹۰٪ توکنِ زمان پاسخ؛ پرسشهای تکراری با کش موجود ≈ صفر |

سند یکبار پردازش میشود اما هزاران بار پرسش میگیرد — صرفهجویی واقعی اینجاست و **دقیقاً همراستا با فلسفه نردبان L0–L4** («گرانترین لایه، آخرین لایه»).

## ۱. جایگاه در معماری دستیار (نردبان بهروزشده)

```
پرسش کاربر (با پرسونا و نقش)
  ▼ L0  پاسخ ساختیافته زنده (شیفت/لوحه/فالت — API ماژولها)        <10ms · ۰ توکن
  ▼ L1  کش معنایی per نقش (موجود)                                  <50ms · ۰ توکن
  ▼ L2  FAQ رسمی تاییدشده (موجود)                                   <80ms · ۰ توکن
  ▼ L3-G  ★ پاسخ گرافی (این سند) — پرسش نقطهای                     <300ms · ~۵۰ توکن
  │       Entity Linking ← پیمایش مسیر گراف ← زیرگراف <100 توکن
  │       ← جملهبندی با مدل اقتصادی محلی + ارجاع سند/صفحه
  ▼ L3-V  بازیابی ترکیبی (بردار + گسترش گراف) + مدل اقتصادی          پرسشهای مبهم/معنایی
  ▼ L4   مدل قوی + خلاصه خوشهها (Global) — پرسشهای تحلیلی کلی      درصد کم
```

**سه مسیر Query Router** (طبق GraphRAG مایکروسافت):

| نوع پرسش | مثال متروی واقعی | مسیر |
| :--- | :--- | :--- |
| نقطهای (Local) | «تعهد مالی پیمانکار در قرارداد نگهداشت پلهبرقی؟» / «آخرین دستورالعمل TSR ناحیه ۲؟» | **L3-G**: پیمایش گراف |
| مبهم/معنایی | «بند مربوط به تأخیر تحویل قطعات» | **L3-V**: بردار ← گسترش گراف |
| کلی/تحلیلی (Global) | «مهمترین ریسکهای همه قراردادهای نگهداشت؟» | **L4**: خلاصه خوشهها |

هر پاسخ L3-G/V پس از تولید، در **کش معنایی L1 همان نقش** ذخیره میشود — پرسش مشابه بعدی حتی به گراف هم نمیرسد.

## ۲. تصمیمهای قطعی فناوری (سازگارشده با پشته پروژه)

نکته یکپارچهسازی: ابزارهای بالغ این حوزه (GLiNER، Hazm، Kùzu، Docling) پایتونیاند و پروژه Node/Next است. تصمیم: **سرویس مجزای «موتور دانش» (Knowledge Engine)** بهصورت sidecar — کاملاً داخلی و روی همان سرور.

| معیار | تصمیم | دلیل |
| :--- | :--- | :--- |
| موتور دانش | **Python 3.11 + FastAPI** (سرویس داخلی، پشت شبکه خصوصی، توکن سرویس) | دسترسی به بهترین ابزار NLP فارسی/گراف |
| ارکستراسیون | **Node (AI Gateway موجود)** — Router، پرسونا، کش L1، RBAC | هیچ منطق محصولی به پایتون نمیرود |
| گراف | **Kùzu توکار** (فایل کنار سرویس) → مهاجرت Neo4j فقط در مقیاس | بدون سرور جدید، شروع چندساعته |
| استخراج | **GLiNER-multilingual** (موجودیت) + **والایش رابطه با Ollama/Qwen2.5 محلی** خروجی JSON-Schema | صفر توکن ابری، مصون از تحریم |
| پیشپردازش فارسی | **Hazm** (نرمالسازی ی/ک، نیمفاصله) + یکسانسازی ارقام + تاریخ شمسی نرمال | بزرگترین ریسک فارسی همینجا حل میشود |
| پارس/OCR | **Docling** + python-docx + **Tesseract fas** برای اسکنها | بخشنامههای اسکنی جا نمیمانند |
| بردار | **LanceDB توکار** در موتور دانش با **همان مدل ۳۸۴بعدی** خانواده موجود (سازگاری امتیاز شباهت با کش L1) | بدون سرور برداری جدید |
| زمانمندی | مدل valid_from/valid_to شمسی از فاز ۰؛ **Graphiti** در فاز ۲ | «آخرین نسخه معتبر» |
| ارزیابی | **RAGAS + Golden Set** ۳۰–۵۰ سواله دستی | بدون سنجش، بهبود ادعاست |

## ۳. هستانشناسی قطعی (Schema-guided — قابل ویرایش از پنل)

استخراج آزاد ممنوع؛ فهرست بسته به مدل داده میشود (دقت↑ تکرار↓ توکن↓). **نوعها در DB نگهداری و از پنل ادمین ویرایش میشوند** — افزودن نوع جدید = بدون کد.

**گرهها (۸ نوع پایه متروی):** `person`(راهبر، مدیر، مسئول OCC) · `org`(معاونت، پیمانکار) · `document`(قرارداد، الحاقیه، بخشنامه، دستورالعمل) · `concept`(ATP، لوحه، TSR) · `equipment`(قطار، پلهبرقی، سیگنالینگ) · `place`(ایستگاه، خط، دپو) · `event`(ابلاغ، حادثه، راهاندازی) · `date`(شمسی نرمال)

**یالها (نمونه):** `مسئول_` · `مدیر_` · `تعهد_مالی` · `اصلاح_میکند` · `ابطال_میکند` · `ابلاغ_به` · `واقع_در` · `مربوط_به` · `رخداد_در`

```
Entity { id, norm_key(شناسه ادغام), name, type, aliases[], accessRoles[] }
REL    { type, source_doc, page, chunk_id, valid_from, valid_to, confidence }
```

<div>🔒 <b>دو قانون تغییرناپذیر:</b> ① یال بدون <code>source_doc</code> ذخیره نمیشود (Provenance اجباری). ② ادغام فقط با <code>norm_key</code>، هرگز نام خام.</div>

## ۴. خط لوله ورود سند (Ingestion — هزینه یکبار)

```
آپلود از پنل «مدیریت دانش» موجود (PDF/Word/TXT/اسکن)
 ① Docling + OCR fas → متن خام + شماره صفحه
 ② Hazm نرمالسازی + قطعهبندی معنایی ~۳۰۰ کلمه
 ③ GLiNER: موجودیتها (فهرست بسته) → Ollama محلی: روابط (JSON مقید، پرامپت بخش ۶)
 ③ب همزمان: embedding هر chunk → LanceDB
 ④ تفکیک هویت سهمرحلهای:
     norm_key یکسان → همان گره
     فازی RapidFuzz ≥۹۰ یا cosine ≥۰.۸۵ → کاندیدای ادغام
     مشکوک → پرسش بله/خیر از LLM محلی → صف تایید انسانی (پنل)
 ⑤ درج در Kùzu با MERGE بر norm_key + Provenance + valid_from
 ⑥ بهروزرسانی خلاصه خوشههای متاثر (برای Global) — تنبل/شبانه
```

- **سند اصلاحی/الحاقیه:** یال `اصلاح_میکند` ساخته میشود و یالهای منسوخِ سند قبلی `valid_to` میگیرند — داده قدیمی حذف نمیشود، تاریخدار میشود.
- وضعیت هر سند در پنل: صف ← OCR ← استخراج ← ادغام ← ایندکسشده (با شمار گره/یال جدید و خطاها).
- **تفکیک هویت فارسی:** «سامانه ایکس / سامانهٔ ایکس / سیستم X» = یک گره با aliases — قلب «ادغام پویا».

## ۵. پاسخگویی مقید (Grounded — ضدتوهم)

- **Entity Linking پرسش:** کلیدواژهها ← norm_key ← گرههای لنگر (fuzzy در نبود تطبیق دقیق).
- پیمایش با فیلترهای اجباری: `valid_to IS NULL` (مگر کاربر تاریخ گذشته بخواهد) + `accessRoles ∩ نقش کاربر` (**دانش غیرمجاز وارد زیرگراف نمیشود** — همان اصل جداسازیِ در بازیابی از سند پرسونا).
- زیرگراف (<۱۰۰ توکن) + پرسش ← مدل اقتصادی با پرامپت مقید (بخش ۶) ← پاسخ + ارجاع.
- **مسیر یافت نشد** ← بدون LLM: «اطلاعات کافی در اسناد یافت نشد» + پیشنهاد نزدیکترین موجودیتها («منظورتان قرارداد ۷ بود؟»).
- در UI: همان `SourceChip` سند UI دستیار — 📖 «الحاقیه ۲، صفحه ۴» ← پنل زمینه، بند را با هایلایت باز میکند. اعتماد کامل حلقه میشود.

### کوئری نمونه — «آخرین تعهد معتبر»
```cypher
MATCH (p:Entity {norm_key:$project})-[r:REL {type:"تعهد_مالی"}]->(v)
WHERE r.valid_to IS NULL
RETURN v.name, r.source_doc, r.page
ORDER BY r.valid_from DESC LIMIT 1
```

## ۷. سناریوهای نهایی

### سناریو ۱ — الحاقیه و «آخرین نسخه» (قاتل داده منسوخ)
دفتر قراردادها الحاقیه ۲ قرارداد نگهداشت را آپلود میکند ← موتور، `اصلاح_میکند` میسازد و تعهد قبلی `valid_to` میگیرد ← مدیر میپرسد «تعهد مالی پیمانکار طبق آخرین الحاقیه؟» ← L3-G: پیمایش با `valid_to IS NULL` ← «۵۰۰ میلیون تومان 📖 الحاقیه ۲، ص۴» در ~۳۰۰ms با ~۵۰ توکن. پاسخ قدیمی هرگز برنمیگردد.

### سناریو ۲ — بخشنامههای جایگزینشونده (دردِ روزمره مترو)
بخشنامه جدید TSR، قبلی را ابطال میکند ← راهبر از دستیار: «سرعت مجاز TSR ناحیه ۲؟» ← گراف فقط بخشنامه معتبر را برمیگرداند با شماره و صفحه ← بار دوم به بعد: L1 کش. (مکمل L0: اگر TSR در ماژول OCC ثبت باشد، L0 جلوتر پاسخ داده — گراف مرجع «اسناد» است، L0 مرجع «داده زنده».)

### سناریو ۳ — پرسش کلی (Global)
مدیر: «مهمترین ریسکهای قراردادهای نگهداشت امسال؟» ← Router: Global ← خلاصه خوشههای قراردادها + مدل قوی ← پاسخ تحلیلی با ارجاع به چند سند — تنها مسیری که L4 میرود، و باز هم مقید.

### سناریو ۴ — صف ادغام هویت
پس از ورود ۳ سند، «شرکت آبانراه» و «پیمانکار آبان راه» کاندیدای ادغام (فازی ۹۳) ← کارشناس دانش در پنل: تایید ← alias ثبت، گراف تمیز میماند — نه دستی از اول، نه ادغام کورکورانه.

### سناریو ۵ — «یافت نشد» صادقانه
«جریمه تأخیر قرارداد ۹؟» — در اسناد نیست ← مسیر گراف خالی ← بدون LLM: «در اسناد یافت نشد؛ نزدیکترین: جریمه تأخیر قرارداد ۷ (📖 ص۱۲). بررسی شود؟» — توهم صفر، مسیر ادامه باز.

### سناریو ۶ — اسکن قدیمی
۴۰ بخشنامه اسکنی ۱۳۹۸ آپلود گروهی ← OCR فارسی ← استخراج شبانه محلی (هزینه ابری صفر) ← صبح: +۸۲۰ گره، +۱٬۴۰۰ یال، ۱۲ کاندیدای ادغام در صف — آرشیو کاغذی، پرسشپذیر شد.

## ۸. پنل مدیریت (افزوده به پنل دستیار — بدونکد)

```
پنل دستیار AI (موجود) + بخش «موتور دانش گرافی»:
├── 📥 صف ورود اسناد        وضعیت مرحلهبهمرحله، خطاها، ورود گروهی، پردازش مجدد
├── 🧬 هستانشناسی           ویرایش انواع گره/رابطه (فهرست بسته) بدون کد + نگاشت رنگ/آیکون
├── 🔗 صف ادغام هویت         کاندیداها با امتیاز + تایید/رد + مدیریت aliases
├── 🕸️ کاوشگر گراف           نمای بصری: جستجوی موجودیت، مسیرها، فیلتر زمان («در تاریخ X»)
├── ⏳ تعارضهای زمانی        یالهای همنوع بدون اصلاح صریح (هشدار به کارشناس)
├── 🎯 ارزیابی               Golden Set editor + اجرای RAGAS + روند سنجهها per نسخه
├── ⚙️ آستانهها              فازی/cosine ادغام، confidence حداقلی، مسیر Router
└── 🔐 دسترسی دانش           accessRoles per سند/دسته (فیلتر در بازیابی — موجود، متصل به گراف)
```

داشبورد دستیار (موجود) دو کارت جدید میگیرد: **نرخ hit مسیر L3-G** و **«توکن صرفهجوییشده توسط گراف»**.

## ۹. ارزیابی — معیار پذیرش هر فاز

| سنجه | هدف |
| :--- | :--- |
| وفاداری به منبع (Faithfulness) | ≥ ۰.۹ |
| صحت پاسخ روی Golden Set (۳۰–۵۰ سوال از بخشنامه/قرارداد واقعی) | ≥ ۰.۸ |
| نرخ «یافت نشد» درست (وقتی واقعاً نیست) | ≥ ۰.۹ |
| دقت استخراج روابط (نمونهگیری دستی) | ≥ ۰.۷۵ |

خطای استخراج به همهجا سرایت میکند — سنجش جدا برای مرحله استخراج الزامی است.

## ۱۱. ریسکها و پاسخ طراحی

| ریسک | پوشش در طرح |
| :--- | :--- |
| کیفیت استخراج فارسی | Hazm + فهرست بسته + والایش LLM محلی + سنجه دقت استخراج + confidence روی یال |
| گرههای تکراری | خط لوله سهمرحلهای هویت + صف تایید انسانی + norm_key |
| داده منسوخ | مدل زمانی از فاز ۰ + `اصلاح/ابطال` + پیشفرض «فقط معتبرِ امروز» |

| پیچیدگی دو زبانه (Node+Python) | مرز تمیز: پایتون فقط موتور دانش با API داخلی؛ محصول/RBAC/کش در Node |
| امنیت اسناد محرمانه | پردازش صددرصد محلی (بدون ابر) + accessRoles در بازیابی + لاگ دسترسی و نسخهبندی گراف |

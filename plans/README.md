## Verification baseline (read before executing any plan)

The repo HAS a working test suite (`npm run test` → Vitest, 22 files) and
typecheck (`npm run typecheck`). Both are used as gates in every plan.

**Windows gotcha**: `npm run build` is defined as
`NODE_OPTIONS=--max-old-space-size=2048 next build`, which fails in
`cmd`/PowerShell because the inline env-var syntax is POSIX. No plan uses
`build` as a gate for this reason — typecheck + test + lint are the gates.
Executors on Windows should not treat a `build` failure as a plan failure.

## Direction — options for the maintainer (not ranked against bugs)

Grounded in repo evidence; each is a choice, not a defect.

1. **Ship the mobile app that already exists.** `mobile/` is a full Expo
   project (App.tsx, eas.json, android/), and README §mobile plus AGENTS.md
   describe native SOS/camera/push flows, yet `src/app/download/page.tsx:138`
   still says "در حال توسعه (به زودی)". The native surface is the biggest
   piece of stated-but-undelivered intent. Trade-off: the API-side auth gaps
   (plans 001/002, backlog JWT-in-URL) should close before a public app store
   release. (Effort: L to finish + release.)

2. **Real notification delivery.** The AI chat history is mocked
   (`ai-chat-interface.tsx:99`) and there is no SMS gateway (the OTP is
   returned in the response as its only delivery path — see plan 002). The data
   model and notification service exist; wiring an SMS/push provider is the
   adjacent-possible step that unblocks password reset and mandatory-bulletin
   push. Trade-off: provider choice/cost; per-provider config already hinted in
   settings (Najva/Pushe). (Effort: M.)

3. **Server-render the read-heavy pages.** With 98/104 pages client-rendered
   and a 2GB build heap, converting the highest-traffic read views
   (dashboard, directory, roster board) to React Server Components would cut
   client JS and the fetch waterfall — the architecture (service layer under
   `src/server/modules`) is already shaped for it. Trade-off: a multi-page
   campaign; do it incrementally, highest-traffic first. (Effort: L.)

## Findings considered and rejected

- **SEC-03 as a standalone "wildcard CORS is exploitable today" bug**:
  downgraded, not rejected — browsers block `*` + credentials, so it is not
  directly exploitable, but the *intent* to allow credentialed cross-origin
  with no allowlist is a real posture problem. Folded into plan 004 as
  hardening rather than an emergency fix.
- **TEST-07 (zero component/UI tests)**: acknowledged as a conscious trade-off
  for a server-dominant app, not a finding needing an immediate plan. Revisit
  with Playwright E2E when the mobile/web surface stabilizes (see backlog).
- **"Build needs 2GB heap" as a bug**: it is a symptom (client-heavy bundle +
  `@xenova/transformers` server bundling), captured under the tech-debt backlog
  (RSC migration, dependency slimming), not a standalone finding.

**نسخه:** 1.0 — سند طراحی بصری و تجربه کاربری (مکمل `AI_ASSISTANT_DESIGN.md`)
**مبنا:** توکن‌ها و اصول `UI_DESIGN.md` پروژه (قرمز متروی خط ۱، Vazirmatn، گرید ۸px، RTL-first، WCAG AA) — گسترش می‌دهد، نقض نمی‌کند.

## ۱) جهت‌گیری طراحی (Design Direction)

**موضوع:** دستیارِ یک محیط عملیاتی ایمنی‌محور است، نه یک چت‌بات سرگرمی. کاربرش راهبری است با دستکش، در نور کم کابین، بین دو اعزام. پس:

> **امضای طراحی: «پاسخِ پیش از پرسش» (Answer-First)**
> برخلاف چت‌بات‌های معمول که با صفحه خالی و یک فیلد شروع می‌شوند، دستیار با **کارت‌های پاسخ آماده‌ی نقش و لحظه‌ی کاربر** باز می‌شود: شیفت بعدی، قطار امروز، آخرین بخشنامه نخوانده — قبل از اینکه چیزی بپرسد. چت، لایه‌ی دوم است.

اصول:
- **اعتماد از شفافیت:** هر پاسخ نشان می‌دهد «از کجا آمده» — چیپ منبع (⚡ داده زنده / 📌 پاسخ رسمی / 📖 آیین‌نامه + استناد / ✨ تولید AI). کاربر عملیاتی باید بداند به چه چیزی تکیه می‌کند.
- **سرعت باید دیده شود:** پاسخ‌های لایه‌های سریع (L0–L2) **بدون انیمیشن تایپ** و یکجا ظاهر می‌شوند (تقلید تایپِ کند برای پاسخ آماده، ضد صداقت است)؛ فقط L3/L4 استریم واقعی دارند.
- **قرمز برند فقط برای هویت و هشدار** — نه برای حباب‌های گفتگو. پالت گفتگو آرام و خنثی می‌ماند.

## ۲) توکن‌های اختصاصی دستیار

```
--ai-bubble-user:    surface (خاکستری روشن) / dark: darkCard     ← حباب کاربر
--ai-bubble-bot:     paper با حاشیه border / dark: ink+border    ← حباب دستیار (بدون رنگ پس‌زمینه سنگین)
--ai-source-live:    #0E9F9F  ⚡ داده زنده سیستم (L0)
--ai-source-faq:     #16A34A  📌 پاسخ رسمی (L2)
--ai-source-doc:     #4F46E5  📖 مستند به آیین‌نامه (L3/L4 با استناد)
--ai-source-gen:     #D97706  ✨ تولید AI بدون منبع مستقیم (کم‌رنگ‌ترین اعتماد)
--ai-tool:           brand-50 پس‌زمینه + brand-600 حاشیه          ← کارت تایید اقدام
```
تایپوگرافی: متن پیام 15/24 (خوانا در کابین)، استناد Caption 12/16، کد/عدد Mono LTR-isolate.

## ۳) UI وب — ساختار سه‌ستونه (ارتقای layout موجود)

```
┌────────────┬──────────────────────────────────────┬───────────────┐
│ ستون راست  │            ناحیه گفتگو                │  ستون چپ      │
│ (RTL: اول) │                                      │  (زمینه)      │
│ ┌────────┐ │  ┌─ هدر: پرسونا فعال + وضعیت ─────┐  │  استنادهای    │
│ │پرسوناها│ │  │ 🤖 دستیار راهبر   ⚡ آماده      │  │  پاسخ جاری:  │
│ │ 🚆راهبر │ │  └────────────────────────────────┘  │  ┌──────────┐ │
│ │ 🛠تعمیر │ │   [کارت‌های Answer-First در شروع]     │  │ آیین‌نامه  │ │
│ ├────────┤ │   💬 حباب‌های گفتگو + چیپ منبع        │  │ ماده ۱۲-۳ │ │
│ │تاریخچه │ │   [کارت تایید اقدام ابزار]            │  │ (پیش‌نمایش)│ │
│ │گفتگوها │ │  ┌────────────────────────────────┐  │  └──────────┘ │
│ │        │ │  │ ＋  بنویسید یا 🎤 بگویید…   ⏎ │  │  اقدامات      │
│ └────────┘ │  └────────────────────────────────┘  │  مرتبط        │
└────────────┴──────────────────────────────────────┴───────────────┘
```

- **ستون راست (اول در RTL):** سوییچر پرسونا (فقط پرسوناهای مجاز نقش) + تاریخچه گفتگوها با جستجو. در <1024px به Drawer تبدیل می‌شود.
- **ستون چپ (زمینه):** وقتی پاسخ استناد دارد، پیش‌نمایش زنده‌ی بند آیین‌نامه همان‌جا باز می‌شود (کلیک روی چیپ استناد در پیام ← هایلایت همان بند) — کاربر بدون ترک گفتگو، منبع را می‌بیند. خالی که باشد: «اقدامات مرتبط» (میان‌برهای ابزار پرسونا).
- ناحیه گفتگو حداکثر عرض 720px (خوانایی سطر فارسی) و وسط‌چین در نمایشگر بزرگ.

### ۳.۱ آناتومی پیام دستیار
```
┌───────────────────────────────────────────┐
│ متن پاسخ …                                │
│                                           │
│ 📖 آیین‌نامه بهره‌برداری، ماده ۱۲-۳  [›]   │ ← چیپ منبع (رنگ per نوع) — کلیک: پنل زمینه
│ ─────────────────────────────────────────│
│ 👍  👎     ⧉ کپی     ↻ پاسخ دقیق‌تر        │ ← اکشن‌بار ظریف (hover در وب، همیشه در موبایل)
└───────────────────────────────────────────┘
```
- «پاسخ دقیق‌تر» = ارتقای دستی به L4 (در سند بک‌اند) — کاربر کنترل دارد، با نشان «مدل قوی» و رعایت سهمیه.
- 👎 ← گزینه‌های سریع دلیل (نادرست/ناقص/بی‌ربط) — خوراک صف بازبینی.

### ۳.۲ کارت تایید اقدام (Tool Confirmation)
```
┌─ 🛠 ثبت فالت جدید ── نیازمند تایید شما ─┐
│ قطار: ۱۲۵ · واگن: ۵ · کد: HVAC-011      │
│ شرح: «تهویه واگن ۵ کار نمی‌کند»          │
│            [ویرایش]   [❌ انصراف]  [✓ ثبت]│
└──────────────────────────────────────────┘
```
پس‌زمینه brand-50 + حاشیه قرمز — تنها عنصر قرمز گفتگو، چون «اقدام واقعی در سیستم» است. پس از تایید: کارت به حالت انجام‌شده با لینک «مشاهده فالت F-1043» جمع می‌شود.

## ۴) UI موبایل (Expo)

### ۴.۰ صفحه شروع — Answer-First
```
┌──────────────────────────────┐
│ 🤖 دستیار راهبر        [⌄]  │ ← سوییچ پرسونا
│ ┌──────────┐ ┌──────────┐   │
│ │⚡ شیفت بعد│ │⚡ قطار من │   │ ← کارت‌های زنده L0 (لمس = پاسخ کامل)
│ │ فردا صبح │ │   ۱۱۸    │   │
│ └──────────┘ └──────────┘   │
│ پرسش‌های پرتکرار نقش شما:    │
│ 〔سرعت TSR؟〕〔ترمز اضطراری؟〕│ ← چیپ‌های FAQ (لمس = پاسخ L2 فوری)
│ ─────────────────────────── │
│ گفتگوهای اخیر …             │
│ ┌──────────────────────────┐│
│ │ ＋   بپرسید…        🎤  ││ ← ورودی چسبان پایین
│ └──────────────────────────┘│
└──────────────────────────────┘
```
- **🎤 ورودی صوتی برجسته** (کاربر با دستکش!) — Speech-to-Text سیستم‌عامل، متن قابل ویرایش قبل از ارسال.
- گفتگو: حباب‌ها تمام‌عرض‌تر، اکشن‌بار همیشه‌نمایان، چیپ استناد ← Bottom Sheet پیش‌نمایش بند (الگوی سه‌لنگری تقویم).
- **حالت آفلاین:** بنر «آفلاین — پاسخ‌های زنده و FAQ ذخیره‌شده در دسترس‌اند»؛ L0 (از کش داده ویجت/تقویم) و FAQهای همگام‌شده کار می‌کنند؛ سوالات جدید صف می‌شوند.
- دسترسی سریع: از هر جای اپ، لمس طولانی روی موجودیت (قطار در لوحه، روز در تقویم) ← «از دستیار بپرس» با زمینه همان موجودیت.

## ۵) رفتار استریم و حالت‌ها (صداقت در حرکت)

| لایه | رفتار نمایش | حس |
| :--- | :--- | :--- |
| L0/L1/L2 | پاسخ **یکجا** ظاهر می‌شود + چیپ منبع، بدون تایپ‌نمایی | «فوری و آماده» |
| L3/L4 | سه‌نقطه تنفس (۱–۲ ثانیه بازیابی) ← استریم توکنی واقعی؛ استنادها در پایان ثابت می‌شوند | «در حال فکر، رو‌راست» |
| ابزار | «در حال خواندن لوحه…» با آیکون همان ماژول ← کارت نتیجه | «دارد کار واقعی می‌کند» |

- دکمه **توقف** حین استریم؛ قطع شبکه وسط استریم ← پیام تا همان‌جا حفظ + «ادامه».
- خطای تأمین‌کننده ← حباب خطای محترمانه + «تلاش دوباره» — هرگز صفحه خالی؛ اگر Gateway به fallback رفت، بی‌سروصدا (کاربر درگیر زیرساخت نمی‌شود).
- **بنر شفافیت** ثابت زیر هدر (کوچک): «پاسخ‌های ✨ تولید هوش مصنوعی‌اند؛ در موارد عملیاتی به آیین‌نامه استناد کنید» — متن از پنل ادمین قابل ویرایش.
- Skeleton، خالی («امروز چه کمکی از من برمی‌آید؟» + کارت‌های Answer-First)، خطا — طبق استاندارد پروژه. `prefers-reduced-motion`: استریم به ظهور بندبند تبدیل می‌شود.

## ۶) سناریوهای تجربه

### سناریو ۱ — پاسخ بدون پرسش
راهبر بین دو اعزام، تب دستیار را باز می‌کند: کارت «⚡ اعزام بعدی: ۱۴:۲۰ از شهرری» همان بالا است. سوالش پاسخ گرفت بدون یک کلمه تایپ. حس: «این ابزار وقت من را می‌فهمد.»

### سناریو ۲ — اعتماد با یک نگاه
«حداکثر سرعت TSR؟» ← پاسخ یکجا با چیپ سبز «📌 پاسخ رسمی» ← راهبر می‌داند این متنِ تاییدشده‌ی سازمان است، نه حدس مدل — بدون کلیک اضافه.

### سناریو ۳ — استناد زنده
سوال پیچیده ترمز ← پاسخ استریمی با چیپ «📖 ماده ۱۲-۳» ← کلیک ← ستون زمینه، بند را با هایلایت باز می‌کند ← کاربر عین متن آیین‌نامه را مقابل پاسخ می‌بیند و به هر دو اطمینان می‌کند.

### سناریو ۴ — صوت با دستکش
تعمیرکار زیر قطار، 🎤 را نگه می‌دارد: «فالت‌های باز قطار صد و بیست و پنج» ← متن پیاده‌شده را می‌بیند، ارسال ← جدول کوچک فالت‌ها در حباب + دکمه «شروع تعمیر HVAC-011» ← کارت تایید ← لمس ✓.

### سناریو ۵ — ارتقای آگاهانه
پاسخ L3 کلی بود ← «↻ پاسخ دقیق‌تر» ← نشان «مدل قوی» و استریم عمیق‌تر ← کاربر هزینه/کیفیت را خودش انتخاب کرد، در چارچوب سهمیه‌ای که ادمین گذاشته.

## ۷) دسترسی‌پذیری و جزئیات

- کنتراست AA در هر دو تم؛ چیپ‌های منبع علاوه بر رنگ، آیکون و متن دارند (کوررنگی).
- ناوبری کیبورد کامل: پیمایش پیام‌ها، ↑ ویرایش آخرین پرسش، فوکوس‌رینگ مرئی؛ aria-live=polite برای استریم.
- هدف لمسی ≥44px (اکشن‌بار موبایل)، ورودی چندخطی با auto-grow تا ۵ خط.
- متن فارسی با اعداد فارسی؛ کد خطا/شماره قطار Mono در جهت LTR ایزوله (بدون شکستن bidi).

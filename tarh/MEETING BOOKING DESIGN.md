# 🗓️ طرح جامع سامانه رزرو وقت جلسه (Meeting Booking Platform)

**نسخه:** 1.0 — سند طراحی و سناریو
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**هدف:** ارتقای ماژول `meetings` فعلی (درخواست ساده pending→approved) به یک **پلتفرم کامل رزرو وقت**: تقویم دردسترس‌بودن هر نقش/مدیر، انواع جلسه قابل تعریف، رزرو اسلاتی شیفت‌آگاه، اتاق جلسه، گردش تایید، یادآوری چندکاناله — و پنل مدیریتی که همه‌ی این‌ها را **بدون کدنویسی** شکل می‌دهد.

---

## ۱) زیرساخت موجود که طرح روی آن سوار می‌شود

- مدل `MeetingRequest` (درخواست‌کننده → مدیر هدف، زمان، مدت، وضعیت، بازبین) — هسته گردش موجود.
- **تقویم شیفت و لوحه** (سند تقویم): منبع حقیقت دردسترس‌بودن — جلسه با شیفت تداخل نکند.
- Notification Gateway (سند اعلان): یادآوری پوش/پیامک. تقویم زندگی: نمایش جلسه به‌عنوان رویداد + ICS.
- الگوهای موجود: گردش تایید `swap`، `settings`، `AuditLog`، RBAC.

**مسئله فعلی:** درخواست‌دهنده زمان را کورکورانه پیشنهاد می‌دهد (نمی‌داند مدیر کی آزاد است)، فقط «جلسه با مدیر» وجود دارد، اتاق و تجهیزات و یادآوری و گزارش نیست.

---

## ۲) مفاهیم کلیدی

| مفهوم | توضیح | مثال |
| :--- | :--- | :--- |
| **نوع جلسه (MeetingType)** | قالب قابل تعریف ادمین: مدت، میزبانِ نقشی، قواعد | «ملاقات مردمی مدیر» ۱۵دقیقه / «جلسه فنی» ۶۰دقیقه |
| **برنامه دردسترس‌بودن (Availability)** | بازه‌های باز هر میزبان/نقش + استثناها | مدیر بهره‌برداری: یکشنبه‌ها ۹–۱۲ |
| **اسلات** | بازه رزروشدنیِ محاسبه‌شده از Availability − شیفت − جلسات − استثنا | یکشنبه ۹:۱۵–۹:۳۰ |
| **میزبان نقشی (Role Host)** | رزرو با «نقش» نه فرد: هر کس آن نقش را دارد و آزاد است | «سرشیفت کشیک ناحیه ۲» |
| **اتاق/منبع** | اتاق جلسه یا منبع (پروژکتور، خودرو) با تقویم خودش | اتاق جلسات دپو |

## ۳) مدل داده (Prisma — گسترش سازگار)

```prisma
model MeetingType {
  id            String  @id @default(cuid())
  key           String  @unique               // "public_visit" | "technical" | ...
  title         String
  description   String?
  durationMin   Int     @default(30)
  bufferMin     Int     @default(0)            // فاصله بین جلسات
  hostMode      String  @default("user")       // user | role  (میزبان فرد یا نقش)
  hostRoleKey   String?                        // اگر role: کدام نقش
  whoCanBook    Json                           // نقش‌های مجاز به رزرو
  approval      String  @default("auto")       // auto | host | secretary  (تایید خودکار/میزبان/دفتر)
  minNoticeHrs  Int     @default(4)            // حداقل فاصله رزرو تا جلسه
  maxPerWeek    Int?                           // سقف رزرو هر کاربر
  needsRoom     Boolean @default(false)
  fields        Json?                          // فیلدهای اضافی فرم رزرو (موضوع، پیوست...)
  color         String?
  isActive      Boolean @default(true)
}

model AvailabilityRule {
  id        String  @id @default(cuid())
  ownerType String                              // user | role
  ownerKey  String                              // userId یا roleKey
  weekday   Int                                 // 0=شنبه ... 6=جمعه (هفته جلالی)
  fromTime  String                              // "09:00"
  toTime    String                              // "12:00"
  typeKeys  Json?                               // محدود به انواع خاص جلسه (اختیاری)
  isActive  Boolean @default(true)

  @@index([ownerType, ownerKey])
}

model AvailabilityException {                   // مرخصی/مأموریت/بلاک دستی
  id        String   @id @default(cuid())
  ownerType String
  ownerKey  String
  date      DateTime
  fromTime  String?
  toTime    String?                             // خالی = کل روز
  reason    String?
  @@index([ownerType, ownerKey, date])
}

model MeetingRoom {
  id       String  @id @default(cuid())
  name     String
  location String?
  capacity Int?
  amenities Json?                               // ["projector","whiteboard"]
  isActive Boolean @default(true)
}

// گسترش MeetingRequest موجود:
model MeetingRequest {
  // ...فیلدهای موجود...
  typeId       String?
  hostRoleKey  String?                          // برای میزبان نقشی
  roomId       String?
  endAt        DateTime?                        // scheduledAt + duration (برای تداخل)
  formData     Json?                            // پاسخ فیلدهای نوع جلسه
  attendees    Json?                            // شرکت‌کنندگان اضافی [userId]
  cancelReason String?
  rescheduleOf String?                          // اگر جابه‌جایی جلسه قبلی است
  remindersSent Json?
  outcomeNote  String?                          // صورت‌جلسه/نتیجه (پس از برگزاری)
}
```

## ۴) موتور اسلات (قلب سیستم)

`GET /api/meetings/slots?typeKey=&hostKey=&date=` اسلات‌های آزاد را محاسبه می‌کند:

```
اسلات‌های خام از AvailabilityRule روز
  − AvailabilityException (مرخصی/بلاک)
  − شیفت‌های میزبان از تقویم شیفت (شب‌کار فردا صبح جلسه نگیرد!)
  − جلسات تاییدشده/در انتظارِ همان بازه (+ buffer)
  − گذشته و کمتر از minNotice
  = اسلات‌های قابل رزرو (برای میزبان نقشی: اجتماع اسلات‌های همه دارندگان نقش،
    با نمایش «اولین فرد آزاد» و توزیع عادلانه round-robin)
```

نکته تمایز: **شیفت‌آگاهی** — هیچ سیستم رزرو عمومی، شیفت گردشی مترو را نمی‌فهمد؛ اینجا دردسترس‌بودن به‌صورت زنده از موتور شیفت/لوحه خوانده می‌شود.

## ۵) سناریوها

### سناریو ۱ — ملاقات مردمی با مدیر (اسلاتی)
ادمین نوع «ملاقات مردمی» می‌سازد: ۱۵دقیقه، میزبان = مدیر بهره‌برداری، تایید = دفتر، یکشنبه‌ها ۹–۱۲. راهبر احمدی در اپ: «رزرو جلسه» ← نوع ← تقویم اسلات‌ها (فقط خانه‌های سبز) ← یکشنبه ۹:۳۰ ← موضوع را می‌نویسد ← رزرو «در انتظار تایید دفتر» ← مسئول دفتر تایید ← هر دو طرف اعلان + رویداد در تقویم زندگی هر دو + یادآوری خودکار ۱ روز و ۱ ساعت قبل.

### سناریو ۲ — میزبان نقشی
کارمند فنی «جلسه با سرشیفت ناحیه ۲» رزرو می‌کند بدون اینکه بداند کدام سرشیفت — سیستم بین سه دارنده نقش، اولین اسلات آزادِ عادلانه را پیشنهاد می‌دهد؛ رزرو به سرشیفتی می‌رسد که آن ساعت شیفت است و جلسه ندارد.

### سناریو ۳ — تداخل شیفتی و جابه‌جایی
مدیر جلسه‌ای ساعت ۱۰ دارد؛ لوحه فردا تغییر می‌کند و شیفتش عوض می‌شود ← سیستم تداخل را می‌بیند ← به هر دو طرف اعلان «نیاز به جابه‌جایی» + سه اسلات جایگزین یک‌کلیکه ← رزرو جدید با `rescheduleOf` به قبلی متصل و تاریخچه حفظ می‌شود.

### سناریو ۴ — اتاق و ظرفیت
«جلسه فنی» با ۵ شرکت‌کننده نیاز به اتاق دارد ← هنگام رزرو، اتاق‌های آزادِ همان بازه با ظرفیت کافی پیشنهاد می‌شود ← اتاق هم مثل میزبان قفل تداخل می‌گیرد.

### سناریو ۵ — لغو و نظم
کاربر ۳ بار در ماه جلسه را لغو آخر‌وقت می‌کند ← قاعده نظم (قابل تنظیم): هشدار و سپس محدودیت رزرو ۲هفته‌ای ← گزارش «نرخ عدم حضور» per نوع جلسه برای مدیر.

## ۶) پنل مدیریت حرفه‌ای و قابل شخصی‌سازی (Admin Console)

```
پنل مدیریت جلسات
├── 📊 داشبورد            جلسات امروز/هفته، نرخ تایید/لغو/عدم حضور، پرترافیک‌ترین میزبان‌ها
├── 🧩 انواع جلسه          ساخت نوع: مدت/بافر/میزبان (فرد یا نقش)/چه کسی رزرو کند/نحوه تایید/
│                          حداقل فاصله/سقف هفتگی/فیلدهای فرم رزرو (فرم‌ساز) — با پیش‌نمایش
├── 🕒 دردسترس‌بودن‌ها      برنامه هفتگی هر میزبان/نقش (گرید بصری کلیک‌وکش) + استثناها
│                          + سوییچ «کسر خودکار شیفت از تقویم» (پیش‌فرض روشن)
├── 🚪 اتاق‌ها و منابع       CRUD اتاق/منبع + تقویم اشغال هر اتاق
├── ✅ کارتابل تایید          تایید/رد/پیشنهاد زمان دیگر (برای انواع نیازمند تایید) — نقش دفتر
├── 🔔 یادآوری‌ها            پله‌های یادآوری per نوع (۱روز/۱ساعت/...)، کانال از Gateway، قالب پیام
├── 📏 قواعد نظم             سقف لغو دیرهنگام، پنجره لغو مجاز، محدودیت خودکار
├── 📈 گزارش‌ها              اکسل جلسات با فیلتر، نرخ عدم حضور، بار جلسات per نقش، خروجی چاپ
├── 👥 دسترسی               چه نقشی چه نوعی را رزرو/میزبانی/مدیریت کند
└── 🗂️ لاگ حسابرسی           هر تغییر پیکربندی و هر تایید/لغو با before/after
```

**Permissionها:** `meetings:book`، `meetings:host`، `meetings:review` (دفتر)، `meetings:view-team`، `meetings-admin:manage`.

## ۷) API (App Router)

```
── کاربر
GET  /api/meetings/types                        ← انواع مجاز برای نقش من
GET  /api/meetings/slots?typeKey=&hostKey=&from=&to=
POST /api/meetings                              ← رزرو {typeKey, slot, formData, attendees}
GET  /api/meetings/my?role=requester|host
POST /api/meetings/[id]/cancel | /reschedule {newSlot} | /outcome {note}
POST /api/meetings/[id]/approve|reject|propose  ← اقدام میزبان/دفتر

── میزبان
GET/POST/PATCH/DELETE /api/meetings/availability      ← برنامه من + استثناها
GET  /api/meetings/host-calendar?from=&to=            ← تقویم جلسات من

── ادمین
GET/POST/PATCH /api/admin/meetings/types | /rooms
GET/PATCH      /api/admin/meetings/availability/[ownerKey]   ← تنظیم برای دیگران
GET/PATCH      /api/admin/meetings/config                     ← یادآوری/نظم/دسترسی
GET            /api/admin/meetings/reports?type=&from=&to=  |  /export
```

## ۸) UI (وب + موبایل)

- **رزرو (کاربر):** انتخاب نوع (کارت‌های آیکون‌دار) ← تقویم ماهانه جلالی با روزهای دارای اسلات ← لیست اسلات‌های روز (چیپ‌های لمسی) ← فرم کوتاه ← تایید. در موبایل: همان الگوی Bottom Sheet تقویم زندگی.
- **میزبان:** «جلسات من» (روز/هفته) + ویرایش دردسترس‌بودن با گرید کلیک‌وکش + اقدام سریع تایید/پیشنهاد از داخل اعلان.
- جلسات به‌صورت لایه در **تقویم زندگی** ظاهر می‌شوند (سند تقویم) + در ICS شخصی.
- همه صفحات: RTL، اعداد فارسی، دارک‌مود، حالات skeleton/خالی/خطا.

## ۱۰) ملاحظات فنی

- **قفل تداخل:** رزرو در تراکنش با بررسی هم‌پوشانی (`scheduledAt/endAt`) روی میزبان و اتاق — دو رزرو همزمان یک اسلات ممکن نیست.
- **هفته جلالی:** weekday با مبدأ شنبه؛ نمایش و ورودی تماماً جلالی، ذخیره ISO.
- موتور اسلات cache کوتاه + ابطال با رویداد (رزرو جدید/تغییر لوحه).
- تست: موتور اسلات (کسر شیفت/استثنا/بافر)، قفل تداخل، round-robin نقش با Vitest.
- حریم خصوصی: درخواست‌دهنده فقط «آزاد/مشغول» میزبان را می‌بیند، نه جزئیات تقویمش.

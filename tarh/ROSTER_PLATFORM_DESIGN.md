# 🚉 طرح جامع لوحه اعزام روزانه نسل بعد (Roster Platform v2)

**نسخه:** 1.0 — سند طراحی و سناریو
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**هدف:** ارتقای ماژول لوحه (قلب عملیات) با چهار محور صریح: **سرعت پاسخ‌گویی بسیار بالا** (نمای شخصی <100ms و آفلاین)، **نمای اختصاصی هر نقش**، **اصلاحیه‌ی زنده‌ی مدیریت‌پذیر** پس از انتشار، و **گزارش‌گیری عملیاتی** — با پنل مدیریتی که قالب‌ها، گردش انتشار، قواعد اعتبارسنجی و دید نقش‌ها را بدون کدنویسی شکل می‌دهد.

---

## ۱) زیرساخت موجود (پایه‌ی قوی فعلی)

- زنجیره داده: `RosterDay` (روز جلالی، وضعیت DRAFT→REVIEW→PUBLISHED→ARCHIVED) ← `RosterVersion` (فایل، checksum، دلیل تغییر) ← `Trip` (ردیف، قطار، جهت تجریش⇄شهرری، ساعات، وضعیت NORMAL/EMPTY/CANCELLED/DEPOT/SPECIAL) ← `TripAssignment` (نقش‌های H1/H2/T/R، تطبیق نام به کاربر).
- `RosterTemplate`: نگاشت ستون‌های اکسل و **نواحی مختصاتی PDF** — پارسر قالب‌پذیر موجود.
- ماژول‌های `diff` (مقایسه نسخه‌ها)، `materialize` (تبدیل به شیفت)، `notes-tasks`، تست‌دار.
- اتصال‌های آماده از اسناد قبلی: تقویم زندگی/ویجت، تعویض شیفت، Notification Gateway (اعلان هدفمند)، Signage (لوحه روی تابلو)، QR قطار، دستیار AI (L0: «قطار من؟»).

**شکاف‌ها:** نمای شخصی با کوئری زنده‌ی چندجدولی (کند در مقیاس)، بی‌آفلاین؛ اصلاح پس از انتشار = آپلود نسخه کامل جدید (سنگین برای یک تغییر کوچک)؛ نام‌های تطبیق‌نشده بی‌گردش؛ اعتبارسنجی و دید نقش‌ها در کد؛ گزارش عملیاتی وجود ندارد.

## ۲) معماری سرعت: «انتشار = پیش‌محاسبه» (پاسخ به سرعت بسیار بالا)

**اصل: هزینه محاسبه یک‌بار در لحظه انتشار پرداخت می‌شود، نه در هر بازدید.**

```
انتشار نسخه لوحه
   │
   ├─▶ ۱. Snapshot روز:        JSON کامل لوحه (همه سفرها+خدمه) در RosterSnapshot
   │        ← نمای عمومی/سرشیفت/OCC = یک SELECT تک‌ردیفی + کش حافظه
   ├─▶ ۲. برش شخصی (Personal):  برای هر کاربرِ حاضر در لوحه، MyRosterDay
   │        {سفرهای من، قطارها، ساعات، نقش، یادداشت‌ها} ← «روز من» = یک ردیف، <100ms
   ├─▶ ۳. Materialize شیفت:     (موجود) + به‌روزرسانی بسته ۳۰روزه ویجت/تقویم
   ├─▶ ۴. ابطال کش و SSE:       رویداد roster.published ← کلاینت‌ها فقط دلتای همان روز را می‌کشند
   └─▶ ۵. اعلان هدفمند:         فقط به افراد حاضر در لوحه (نه broadcast) از Gateway
```

- **آفلاین موبایل:** «روز من + ۷ روز آینده» پس از هر همگام‌سازی در دستگاه ذخیره می‌شود — راهبر در تونل هم لوحه‌اش را دارد (با نشان زمان آخرین همگام‌سازی).
- **ETag/If-None-Match** روی snapshot: کلاینت‌ها ۳۰۴ می‌گیرند مگر نسخه عوض شده باشد.
- اصلاحیه (بخش ۴) فقط دلتا را دوباره پیش‌محاسبه می‌کند — نه کل روز.

```prisma
model RosterSnapshot {
  id          String   @id @default(cuid())
  rosterDayId String   @unique
  versionNo   Int
  payload     Json                     // لوحه کامل نرمال‌شده برای نمایش
  etag        String
  builtAt     DateTime @default(now())
}

model MyRosterDay {                    // برش شخصی — کلید سرعت
  id         String  @id @default(cuid())
  userId     String
  jalaliDate String
  versionNo  Int
  payload    Json                      // سفرهای من + متادیتا
  @@unique([userId, jalaliDate])
  @@index([jalaliDate])
}
```

## ۳) نمای هر نقش (پاسخ به «برای هر نقش»)

| نقش | نما | محتوا |
| :--- | :--- | :--- |
| **راهبر/خدمه** | «روز من» | فقط سفرهای خودش: قطار، جهت، اعزام/رسید، هم‌خدمه، یادداشت سفر؛ شمارش معکوس اعزام بعدی؛ دکمه «فالت این قطار» (QR-less) |
| **سرشیفت ناحیه** | «تابلوی ناحیه» | سفرهای ناحیه‌اش + پوشش خدمه (کمبود قرمز)، تطبیق‌نشده‌ها، غایبان (اتصال حضور)، اقدام: جایگزینی سریع |
| **OCC** | «خط زمان زنده» | گانت سفرها روی محور ساعت، الان-خط قرمز، وضعیت‌ها رنگی، اصلاحیه‌های امروز، اتصال TSR |
| **برنامه‌ریز/دفتر** | «میز انتشار» | آپلود/پارس/اعتبارسنجی/دیف/انتشار + ویرایش درجا + اصلاحیه |
| **مدیر** | «داشبورد + گزارش» | KPIهای روز/ماه، مقایسه، خروجی‌ها |

دید هر نقش (کدام ستون‌ها، کدام نواحی، مشاهده نام کامل یا فقط نقش هم‌خدمه) در **ماتریس دید پنل** تنظیم می‌شود — نه در کد.

## ۴) اصلاحیه زنده (Amendment) — مدیریت تغییرات پس از انتشار

به‌جای «نسخه کامل جدید» برای هر تغییر کوچک:

```prisma
model RosterAmendment {
  id          String   @id @default(cuid())
  rosterDayId String
  seq         Int                       // اصلاحیه شماره N روز
  kind        String                    // trip_time | trip_status | crew_replace | trip_add | trip_remove | note
  targetTripId String?
  before      Json?
  after       Json
  reason      String                    // اجباری
  status      String   @default("draft") // draft | approved | applied
  createdBy   String
  approvedBy  String?
  appliedAt   DateTime?
  @@index([rosterDayId, seq])
}
```

- برنامه‌ریز/سرشیفتِ مجاز، تغییر را درجا ثبت می‌کند (مثلاً «جایگزینی H1 سفر ۴۷») ← تایید (قابل تنظیم: کدام نوع اصلاحیه نیاز به تایید چه نقشی دارد) ← اعمال اتمیک: به‌روزرسانی Trip/Assignment + دلتای Snapshot و MyRosterDay فقط برای افراد متاثر + **اعلان فقط به همان افراد** («سفر ۱۴:۲۰ شما لغو شد — اصلاحیه ۳») + رنگ «تغییرکرده» در نمای همه.
- تاریخچه کامل: نسخه پایه + زنجیره اصلاحیه‌ها = بازسازی لوحه در هر لحظه از روز (برای بررسی سوانح: «ساعت ۱۵:۰۰ لوحه چه بود؟»).
- diff موجود ارتقا می‌یابد: نسخه‌به‌نسخه + اصلاحیه‌به‌اصلاحیه.

## ۵) اعتبارسنجی و تطبیق (کیفیت قبل از انتشار)

**موتور قواعد قابل پیکربندی** — هر قاعده: شدت (خطا=مسدود / هشدار)، روشن/خاموش، پارامتر:
- خدمه ناقص (سفر NORMAL بدون H1)، شخص هم‌زمان در دو سفر، فاصله استراحت کمتر از حد بین دو سفر یک نفر، تطبیق‌نشده‌ها بالای آستانه، توالی ساعات نامعتبر، قطار خارج سرویس در ناوگان (اتصال به سند فالت!)، شخص در مرخصی تاییدشده (اتصال به فرم‌ها).
- **صف تطبیق نام:** rawName های نامطمئن با پیشنهاد fuzzy (top-3 + امتیاز) ← تایید یک‌کلیکه ← ذخیره در **دیکشنری املای اسامی** (alias) برای همیشه — هر هفته دقیق‌تر.

## ۶) گزارش‌گیری عملیاتی (پاسخ به «قابل گزارش‌گیری»)

همه با فیلتر بازه جلالی/قطار/فرد/ناحیه + **خروجی اکسل دوشیته و چاپ**:
- **کارکرد خدمه:** تعداد سفر/ساعت per نفر و ماه (پایه اضافه‌کار و عدالت توزیع)، تفکیک نقش H1/H2/T/R.
- **کارکرد ناوگان:** سفر per قطار (تقاطع با گزارش فالت: قطار پرکار/پرخرابی)، سفرهای EMPTY/DEPOT.
- **پایداری برنامه:** تعداد اصلاحیه per روز/نوع/دلیل، لغوها، جایگزینی‌های اضطراری — سنجه کیفیت برنامه‌ریزی.
- **انتشار:** ساعت انتشار هر روز (تأخیرها)، فاصله انتشار تا اولین اعزام.
- **پوشش:** نرخ تطبیق نام، سفرهای بی‌خدمه در لحظه انتشار.
- داشبورد مدیر: امروز در یک نگاه + روند ماهانه + مقایسه با ماه قبل.

## ۷) سناریوها

### سناریو ۱ — صبح راهبر (سرعت + آفلاین)
احمدی ۰۵:۳۰ در راه دپو، در تونل مترو بدون آنتن، اپ را باز می‌کند: «روز من» از کش دستگاه در لحظه — قطار ۱۱۸، اعزام ۰۶:۱۵، هم‌خدمه، یادداشت «TSR ناحیه ۲». شمارش معکوس: ۴۵ دقیقه. (همین داده در ویجت صفحه اصلی هم هست.)

### سناریو ۲ — انتشار روز
برنامه‌ریز ۲۲:۰۰ فایل PDF لوحه فردا را می‌کشد و رها می‌کند ← قالب PDF ذخیره‌شده، ۲۱۰ سفر را می‌خواند ← اعتبارسنجی: ۲ خطا (تداخل شخص، خدمه ناقص) و ۵ نام نامطمئن ← اصلاح درجا + تایید تطبیق‌ها ← دیف با پیش‌نویس قبلی ← انتشار ← پیش‌محاسبه ۳ ثانیه ← اعلان هدفمند به ۱۸۷ نفرِ حاضر در لوحه ← تابلوهای Signage دپو به‌روز.

### سناریو ۳ — اصلاحیه ظهر (هدفمند، نه طوفان اعلان)
راننده سفر ۱۴:۲۰ بیمار می‌شود ← سرشیفت از تابلوی ناحیه: «جایگزینی خدمه» ← پیشنهاد سیستم: افراد آزادِ واجد نقش همان بازه (از تقویم/حضور) ← انتخاب رضایی ← تایید دفتر ← اصلاحیه ۲ اعمال ← فقط ۳ نفر متاثر اعلان می‌گیرند؛ «روز من» رضایی و ویجتش در لحظه به‌روز؛ OCC ردیف را نارنجی «تغییرکرده» می‌بیند.

### سناریو ۴ — بررسی پس از رخداد
واحد ایمنی می‌پرسد «ساعت ۱۵:۰۰ سه‌شنبه، خدمه سفر ۵۱ چه کسانی بودند؟» ← نمای بازسازی تاریخی: نسخه پایه + اصلاحیه‌های تا آن ساعت ← پاسخ قطعی با استناد به شماره اصلاحیه و تاییدکننده.

### سناریو ۵ — گزارش ماهانه عدالت
مدیر: گزارش کارکرد مرداد ← جدول per نفر: سفر/ساعت/سهم شب ← مرتب‌سازی نشان می‌دهد دو نفر ۳۰٪ بالای میانگین‌اند ← اکسل به واحد برنامه‌ریزی برای توازن شهریور.

## ۸) پنل مدیریت حرفه‌ای و قابل شخصی‌سازی (Admin Console)

```
پنل مدیریت لوحه
├── 📊 داشبورد            وضعیت امروز/فردا، سرعت انتشار، اصلاحیه‌ها، تطبیق‌نشده‌ها
├── 🖇️ قالب‌های فایل         (ارتقای موجود) طراح بصری: پیش‌نمایش فایل نمونه + کشیدن قاب روی
│                          نواحی PDF / انتخاب ستون اکسل با کلیک ← تست پارس زنده روی همان فایل
│                          + چند قالب فعال (تشخیص خودکار از checksum/ساختار)
├── 🔀 گردش انتشار           مراحل (پیش‌نویس/بازبینی/انتشار) و نقش هر مرحله؛ انتشار زمان‌بندی‌شده؛
│                          قفل ساعت (مثلاً پس از ۲۳:۰۰ فقط با نقش ویژه)
├── ✅ قواعد اعتبارسنجی       هر قاعده: روشن/خاموش، شدت، پارامتر (حداقل استراحت=۳۰د...) — بدون کد
├── ✏️ قواعد اصلاحیه          کدام نوع اصلاحیه ← نیازمند تایید کدام نقش؛ مهلت اصلاح؛ دلایل استاندارد
├── 👥 ماتریس دید نقش‌ها      نقش × (ستون‌ها/نواحی/نام هم‌خدمه/یادداشت‌ها) + نمای پیش‌فرض هر نقش
├── 🧑‍🤝‍🧑 تطبیق اسامی           صف تایید + دیکشنری alias + آستانه اطمینان خودکار
├── 🔔 اعلان‌ها               نگاشت رویدادها (انتشار/اصلاحیه/لغو سفر) به قواعد Gateway per شدت
├── 🖥️ نمایش عمومی            پیکربندی لوحه روی Signage (کدام ستون‌ها، ناحیه هر تابلو)
├── 🗄️ بایگانی و نگهداری      سیاست آرشیو، دوره نگهداری snapshot/اصلاحیه، خروجی بایگانی سالانه
├── 📈 مرکز گزارش             گزارش‌های بخش ۶ + اکسل/چاپ + گزارش ماهانه خودکار
└── 🗂️ لاگ حسابرسی            هر انتشار/اصلاحیه/تغییر پیکربندی با before/after
```

**Permissionها:** `roster:view-own`، `roster:view-region`، `roster:view-all`، `roster:upload`، `roster:publish`، `roster:amend` (per نوع)، `roster:approve-amendment`، `roster:match-names`، `roster:reports`، `roster-admin:manage`.

## ۱۰) UI (وب + موبایل — چکیده)

- **«روز من» (موبایل):** کارت‌های سفر به ترتیب ساعت، سفر بعدی بزرگ با شمارش معکوس، رنگ وضعیت، نشان «اصلاح‌شده»، کشیدن به پایین = همگام‌سازی. همان الگوی Bottom Sheet تقویم برای جزئیات سفر.
- **تابلوی ناحیه (سرشیفت، وب/تبلت):** جدول متراکم با ستون پوشش، ردیف‌های مسئله‌دار بالا، اقدام درجا.
- **خط زمان OCC:** گانت افقی، الان-خط، زوم بازه، کلیک سفر = پنل جزئیات + اصلاحیه‌ها.
- **میز انتشار:** سه‌مرحله‌ای (فایل ← اعتبارسنجی/تطبیق ← دیف و انتشار) با نوار پیشرفت؛ طراح قالب با پیش‌نمایش زنده.
- استانداردهای پروژه: RTL، اعداد فارسی، جلالی، دارک‌مود، skeleton/خالی/خطا.

## ۱۲) ملاحظات فنی

- پیش‌محاسبه در background پس از تراکنش انتشار؛ idempotent با versionNo؛ شکست = retry بدون خراب‌کردن نسخه.
- اصلاحیه‌ها اتمیک (تراکنش: Trip + Snapshot delta + MyRosterDay متاثر + Outbox اعلان).
- ماتریس دید در **سرور** روی payload اعمال می‌شود (نه فیلتر UI).
- سازگاری عقب‌رو: APIهای فعلی حفظ؛ مسیرهای سریع جدید در کنارشان؛ materialize شیفت دست‌نخورده.
- تست: پیش‌محاسبه/دلتا، بازسازی تاریخی، قواعد اعتبارسنجی پارامتری، ماتریس دید — Vitest (کنار service.test موجود).

# 🔔 طرح جامع سامانه اعلان چندکاناله مقاوم به تحریم (Notification Gateway)

**نسخه:** 1.0 — سند طراحی و سناریو
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**هدف:** ارتقای ماژول `notifications` فعلی به یک **دروازه اعلان چندکاناله** که کاملاً روی زیرساخت داخلی ایران کار کند — پوش موبایل با **درایور سرویس ایرانی (پوشه / نجوا)** و **درایور Self-hosted**، انتخاب و پیکربندی کامل توسط مدیر از پنل، بدون یک خط کدنویسی مجدد.

---

## ۱) چشم‌انداز و اصول طراحی

زیرساخت موجود که طرح روی آن سوار می‌شود: مدل `Notification` (اعلان درون‌برنامه‌ای)، کانال بلادرنگ SSE (`src/server/realtime/bus.ts`)، ماژول `settings`، `AuditLog`، و الگوی موفق **Gateway با Circuit Breaker** از ماژول AI.

**اصول:**
1. **هیچ وابستگی به FCM/APNs به‌عنوان ستون اتکا** — سرویس‌های آمریکایی در معماری حیاتی جایی ندارند.
2. **درایورمحور (Driver-based):** پوش از پشت یک интерфейс واحد ارسال می‌شود؛ مدیر در پنل انتخاب می‌کند کدام درایور فعال باشد (پوشه، نجوا، Self-hosted) — افزودن درایور جدید در آینده فقط یک کلاس جدید است، نه بازنویسی.
3. **تحویل تضمینی چندلایه:** درون‌برنامه (SSE) → پوش (درایور فعال) → پیامک داخلی برای پیام‌های حیاتی. اگر لایه‌ای شکست خورد، لایه بعدی خودکار فعال می‌شود.
4. **همه‌چیز داده‌محور:** کانال هر رویداد، قالب پیام، سهمیه، ساعت سکوت — همه از پنل، هیچ‌چیز در کد.

## ۳) درایورهای پوش (قلب طرح)

هر درایور یک کلاس با интерфейс مشترک است:

```ts
interface PushDriver {
  key: 'pushe' | 'najva' | 'selfhosted';
  registerDevice(userId, token, platform): Promise<void>;
  send(msg: PushMessage, targets: DeviceTarget[]): Promise<DeliveryReport>;
  healthCheck(): Promise<DriverHealth>;   // برای Circuit Breaker و چراغ پنل
}
```

### ۳.۱ درایور «پوشه» (Pushe)
- SDK رسمی اندروید (+ وب‌پوش)؛ سرورها داخل ایران — مصون از تحریم و فیلترینگ.
- ارسال از سرور با REST API پوشه (توکن API در پنل وارد می‌شود).
- قابلیت‌های بومی: ارسال بر اساس تگ/سگمنت، زمان‌بندی، گزارش تحویل و کلیک.
- در اپ Expo: از طریق Config Plugin / ماژول native اضافه می‌شود (نیازمند EAS Build — یک‌بار).

### ۳.۲ درایور «نجوا» (Najva)
- ساختار مشابه پوشه؛ SDK اندروید و وب‌پوش، REST API سروری، پنل داخلی.
- وجود دو درایور ایرانی = **قدرت چانه‌زنی و مهاجرت یک‌کلیکه**: اگر کیفیت/قیمت یکی افت کرد، مدیر از پنل سوئیچ می‌کند؛ دستگاه‌ها در هر دو قابل ثبت‌اند (dual-register اختیاری برای مهاجرت نرم).

### ۳.۳ درایور Self-hosted (استقلال کامل)
برای سازمانی که می‌خواهد **هیچ** طرف سومی در مسیر پیام نباشد (اقتضای امنیتی مترو):
- **هسته:** یک سرور سبک WebSocket/SSE اختصاصی (همان `bus.ts` تعمیم‌یافته) یا استقرار **ntfy** روی سرور داخلی سازمان.
- **اندروید:** Foreground Service پایدار در اپ Expo (ماژول native کوچک) که اتصال دائم به سرور نگه می‌دارد؛ با WorkManager بعد از ریبوت/کیل شدن احیا می‌شود.
- **مصرف باتری:** اتصال با heartbeat تطبیقی (۳۰ثانیه تا ۵دقیقه بسته به شارژ/شبکه). در پنل قابل تنظیم است که این درایور **فقط برای نقش‌های حیاتی** (سرشیفت، OCC، مدیر بحران) فعال باشد و بقیه روی درایور ایرانی بمانند — بهترین تعادل.
- **مزیت:** پیام SOS و بحران از شبکه داخلی سازمان هم عبور می‌کند (حتی در قطعی اینترنت کشوری، روی اینترانت).

### ۳.۴ زنجیره Fallback و Circuit Breaker
- هر درایور شمارنده خطا دارد؛ ۵ خطای متوالی ← مدار باز ← درایور بعدی زنجیره (ترتیب را مدیر می‌چیند: مثلاً پوشه ← Self-hosted).
- health check دوره‌ای، مدار را می‌بندد. همه رویدادها در پنل «سلامت کانال‌ها» با چراغ سبز/زرد/قرمز دیده می‌شوند.
- پیام‌های شکست‌خورده به صف Outbox برمی‌گردند (تلاش مجدد با backoff تا سقف قابل تنظیم).

## ۴) لایه پیامک (پیام‌های حیاتی)

- درایورهای SMS داخلی: **کاوه‌نگار / ملی‌پیامک / SMS.ir** (همان интерфейс درایور، انتخاب در پنل + کلید API).
- فقط برای رویدادهایی که مدیر «حیاتی» علامت زده (SOS، بخشنامه اجباری، نقض SLA بحرانی) و معمولاً وقتی پوش/درون‌برنامه ظرف N دقیقه **دیده نشده**.
- سهمیه‌بندی: سقف روزانه کل و per کاربر در پنل + گزارش هزینه.

## ۵) مدل داده (Prisma Schema)

```prisma
model NotificationDevice {
  id         String   @id @default(cuid())
  userId     String
  platform   String                    // android | ios_pwa | web
  driver     String                    // pushe | najva | selfhosted
  token      String                    // توکن SDK یا شناسه اتصال self-hosted
  deviceName String?
  lastSeenAt DateTime @default(now())
  isActive   Boolean  @default(true)
  createdAt  DateTime @default(now())

  @@unique([driver, token])
  @@index([userId, isActive])
}

model NotificationTemplate {
  id        String  @id @default(cuid())
  eventKey  String  @unique            // "roster.published" | "sos.triggered" | "form.approved" ...
  title     String                     // با متغیر: «لوحه {date} منتشر شد»
  body      String
  smsText   String?                    // نسخه کوتاه پیامکی
  link      String?                    // deep link
  isActive  Boolean @default(true)
  updatedBy String?
  updatedAt DateTime @updatedAt
}

model NotificationRule {
  id           String  @id @default(cuid())
  eventKey     String  @unique
  severity     String  @default("normal")   // info | normal | important | critical
  channels     Json                          // ["inapp","push"] یا ["inapp","push","sms"]
  audience     Json?                         // پیش‌فرض گیرندگان: نقش‌ها/گروه‌ها (رویداد می‌تواند override کند)
  smsIfUnseenMinutes Int?                    // پیامک اگر تا N دقیقه دیده نشد (فقط critical)
  respectQuietHours  Boolean @default(true)  // ساعت سکوت (شب‌کارها استثنا می‌شوند بر اساس شیفت!)
  isActive     Boolean @default(true)
}

model NotificationOutbox {
  id          String   @id @default(cuid())
  eventKey    String
  userId      String
  channel     String                    // inapp | push | sms
  driver      String?                   // درایور استفاده‌شده
  payload     Json
  status      String   @default("queued")  // queued | sent | delivered | seen | failed | expired
  attempts    Int      @default(0)
  lastError   String?
  sentAt      DateTime?
  deliveredAt DateTime?
  seenAt      DateTime?
  createdAt   DateTime @default(now())

  @@index([status, createdAt])
  @@index([userId, eventKey])
}

model NotificationPreference {
  id       String @id @default(cuid())
  userId   String @unique
  channels Json                         // ترجیح کاربر per دسته رویداد (در چارچوب اجباری‌های سازمان)
  quietHours Json?                      // {from:"23:00", to:"07:00"} — برای رویدادهای غیرحیاتی
}
```

> `Notification` موجود دست‌نخورده می‌ماند (رکورد درون‌برنامه‌ای)؛ Outbox لایه تحویل چندکاناله را اضافه می‌کند. **حلقه دیده‌شدن:** باز شدن اعلان در اپ ← `seenAt` ← پیامک پشتیبان لغو می‌شود.

## ۶) سناریوهای کاربری

### سناریو ۱ — انتشار لوحه (رویداد عادی)
ادمین لوحه فردا را منتشر می‌کند ← Gateway قاعده `roster.published` را می‌خواند: کانال‌ها = درون‌برنامه + پوش، مخاطب = پرسنل حاضر در لوحه ← قالب «لوحه {date} منتشر شد — شیفت شما: {shift}» رندر می‌شود ← SSE برای آنلاین‌ها، پوش از **درایور فعال (پوشه)** برای بقیه. گزارش تحویل در پنل: ۱۸۷ ارسال، ۱۸۲ تحویل، ۱۴۱ دیده‌شده.

### سناریو ۲ — SOS (رویداد حیاتی، نمایش قدرت چندلایه)
راهبر SOS می‌زند ← قاعده `sos.triggered`: severity=critical، کانال‌ها = هر سه، ساعت سکوت نادیده، مخاطب = OCC و مدیر بحران ← همزمان: SSE (پاپ‌آپ قرمز تمام‌صفحه در OCC)، پوش از **درایور Self-hosted** (نقش‌های حیاتی روی این درایورند — حتی روی اینترانت می‌رسد)، و چون critical است پیامک **بدون انتظار** هم می‌رود. هر سه در Outbox ردیابی می‌شوند.

### سناریو ۳ — سوئیچ درایور بدون کد
سازمان تصمیم می‌گیرد از پوشه به نجوا مهاجرت کند. مدیر در پنل: نجوا را «فعال» و پوشه را «فقط دریافت» می‌کند ← اپ در اجرای بعدی توکن نجوا را هم ثبت می‌کند (dual-register) ← دو هفته بعد، مدیر پوشه را خاموش می‌کند. **صفر تغییر کد، صفر انتشار نسخه.**

### سناریو ۴ — قطعی سرویس ایرانی
API پوشه ۵ خطای متوالی می‌دهد ← Circuit Breaker مدار را باز می‌کند ← پیام‌ها خودکار به درایور بعدی زنجیره (Self-hosted) می‌روند ← چراغ پوشه در پنل قرمز + اعلان به ادمین فنی ← health check هر ۲ دقیقه؛ با برگشت سرویس، مدار بسته و ترافیک برمی‌گردد.

### سناریو ۵ — بخشنامه اجباری با پیگیری پیامکی
بخشنامه ایمنی «الزامی» منتشر می‌شود ← درون‌برنامه + پوش ← قاعده: `smsIfUnseenMinutes=60` ← ۶۰ دقیقه بعد، جاب Outbox کاربرانی که `seenAt` ندارند را برمی‌دارد و فقط برای آن‌ها پیامک می‌فرستد ← گزارش «چه کسی هنوز ندیده» برای پیگیری سرشیفت.

### سناریو ۶ — احترام به شب‌کار
اعلان غیرحیاتی «نتیجه نظرسنجی» ساعت ۱۰ صبح صادر می‌شود ← کاربری که شیفت شب بوده و در ساعت سکوتش است، پوش نمی‌گیرد؛ اعلان در مرکز اعلان می‌ماند و پوش در پایان ساعت سکوت (تعریف‌شده بر اساس شیفت واقعی‌اش از تقویم!) ارسال می‌شود.

## ۷) پنل مدیریت حرفه‌ای و قابل شخصی‌سازی (Admin Console)

فلسفه: **مدیر بدون برنامه‌نویس، کل رفتار اعلان سازمان را شکل می‌دهد.**

```
پنل مدیریت اعلان‌ها
├── 📊 داشبورد تحویل        ارسال/تحویل/دیده‌شدن per کانال و رویداد، هزینه پیامک، روند
├── 🔌 درایورها و کانال‌ها    ◉ انتخاب درایور پوش فعال (پوشه/نجوا/Self-hosted)
│                            کلیدهای API، تست اتصال یک‌کلیکه، زنجیره Fallback (Drag & Drop)،
│                            نگاشت نقش←درایور (نقش‌های حیاتی ← Self-hosted)
├── 🩺 سلامت کانال‌ها         چراغ زنده هر درایور، وضعیت Circuit Breaker، تاریخچه قطعی
├── 📝 قالب‌های پیام          ویرایش عنوان/متن/پیامک هر رویداد با متغیرها + پیش‌نمایش زنده
│                            + ارسال آزمایشی به خودم
├── 🔀 قواعد مسیریابی         per رویداد: شدت، کانال‌ها، مخاطب پیش‌فرض، پیامک‌اگر‌دیده‌نشد،
│                            احترام به ساعت سکوت — جدول قواعد با ویرایش درجا
├── 📱 دستگاه‌ها              دستگاه‌های ثبت‌شده هر کاربر، ابطال دستگاه مشکوک
├── 💰 سهمیه و هزینه          سقف پیامک روزانه/ماهانه، هشدار آستانه، گزارش هزینه per رویداد
├── ⏰ ساعت سکوت سازمانی      پیش‌فرض‌ها + استثنای خودکار شیفت (اتصال به تقویم شیفت)
├── 📣 ارسال دستی/کمپین       پیام یک‌باره به نقش/گروه/فرد با انتخاب کانال + زمان‌بندی
└── 🗂️ لاگ و Outbox           جستجوی هر پیام: چه شد، کدام درایور، چرا شکست، تلاش مجدد دستی
```

**نکات کلیدی پنل:**
- **انتخاب درایور = یک رادیوباتن + کلید API + دکمه «تست اتصال»** که یک پوش واقعی به گوشی مدیر می‌فرستد.
- تغییر هر قاعده/قالب/درایور در `AuditLog` با before/after ثبت می‌شود؛ permission جدا: `notifications-admin:manage`.
- **Presetها:** سه پروفایل آماده («فقط داخلی»، «متعادل»، «حداکثر اطمینان») که زنجیره‌ها و قواعد را یک‌کلیکه می‌چینند.

## ۱۰) ملاحظات فنی

- **iOS:** بدون APNs پوش native ممکن نیست؛ مسیر iOS = PWA Web Push (iOS 16.4+) از طریق درایور وب همان سرویس ایرانی + پیامک برای حیاتی‌ها — در پنل به‌صورت شفاف «پوشش iOS» نمایش داده می‌شود.
- **امنیت:** توکن‌ها فقط سمت سرور؛ payload پوش حداقلی (عنوان + deep link — داده حساس فقط بعد از باز شدن اپ و احراز هویت بارگیری می‌شود).
- **تست:** Gateway، زنجیره fallback و قواعد مسیریابی با Vitest (mock درایورها)؛ الگوی تست AI Gateway موجود.
- **کارایی:** ارسال گروهی batch به API درایورها؛ جاب Outbox با backoff نمایی؛ ایندکس‌های تعریف‌شده.
- **حریم خصوصی:** شماره موبایل فقط برای پیامک حیاتی سازمانی؛ کاربر در ترجیحات، کانال‌های غیرالزامی را کنترل می‌کند.

# 📣 طرح جامع سامانه اطلاع‌رسانی و اعلانات سازمانی (Announcements Platform)

**نسخه:** 1.0 — سند طراحی و سناریو
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**هدف:** ارتقای ماژول محتوا/اطلاع‌رسانی فعلی (`content` / `pa` / مدل `Post`) به یک **پلتفرم کامل اعلانات سازمانی**: انواع اطلاعیه با مخاطب هدفمند، بنر و اعلان فوری تمام‌صفحه، تابلوی دیجیتال ایستگاه/دپو (Signage)، تاییدخوانی ردیابی‌پذیر، و پنل مدیریت کاملاً بدون‌کد.

**مرز با سند اعلان (Notification):** آن سند «**کانال تحویل**» است (SSE/پوش/پیامک)؛ این سند «**محتوا و نمایش**» است. اتصال: انتشار هر اطلاعیه، رویداد `announcement.published` را به همان Notification Gateway می‌فرستد — دوباره‌کاری صفر.

---

## ۱) زیرساخت موجود که طرح روی آن سوار می‌شود

- مدل `Post` با گردش `draft → review → approved → published`، انتشار زمان‌بندی‌شده (`publishAt`)، پرچم `mandatory`، دسته‌بندی و کاور.
- `PostRead` (رسید خواندن)، `PostReaction`، `Comment` — پایه تعامل.
- الگوی اثبات‌شده «بخشنامه ایمنی» (مودال اجباری + رسید immutable با دستگاه/زمان/موقعیت) — برای اطلاعیه‌های الزامی بازاستفاده می‌شود.
- صفحه `pa` (اطلاع‌رسانی عمومی)، صفحه‌ساز پویا (UI Builder) برای ابزارک داشبورد، و ماژول `settings` + `AuditLog`.

---

## ۲) مفاهیم و انواع اعلان سازمانی

| نوع | رفتار نمایش | نمونه |
| :--- | :--- | :--- |
| **خبر (News)** | فید اخبار + کارت داشبورد | «افتتاح رختکن جدید دپو» |
| **اطلاعیه (Notice)** | فید + نشان در مرکز اعلان | «تغییر ساعت سرویس دهه اول محرم» |
| **اطلاعیه الزامی (Must-Read)** | مودال غیرقابل‌رد تا تایید + رسید | «دستورالعمل جدید توقف اضطراری» |
| **بنر فوری (Urgent Banner)** | نوار قرمز بالای همه صفحات وب/موبایل تا انقضا | «اختلال ارتباط رادیویی ناحیه ۳» |
| **اعلان بحران (Emergency)** | تمام‌صفحه قرمز + صدا، عبور از ساعت سکوت | «تخلیه ایستگاه X» |
| **تابلو (Signage)** | نمایش تمام‌صفحه چرخشی روی مانیتور ایستگاه/دپو | برنامه روز + اطلاعیه‌ها |

هر نوع، سطوح نمایش (Surface) خودش را دارد: فید، ابزارک داشبورد، بنر، مودال، مرکز اعلان، تابلو — و مدیر per اطلاعیه سطح‌ها را تیک می‌زند.

## ۳) مدل داده (گسترش Prisma — سازگار با Post موجود)

```prisma
// گسترش Post با فیلدهای هدف‌گیری و نمایش (migration افزایشی)
model Post {
  // ...فیلدهای موجود...
  kind          String   @default("news")     // news | notice | must_read | urgent_banner | emergency
  audience      Json?                          // {roles:[], groups:[], stations:[], shiftCodes:[], userIds:[]} — خالی = همه
  surfaces      Json?                          // ["feed","dashboard","banner","modal","signage"]
  priority      Int      @default(0)           // ترتیب/سنجاق
  pinnedUntil   DateTime?                      // سنجاق در بالای فید
  expiresAt     DateTime?                      // پایان نمایش (بنر/تابلو)
  ackRequired   Boolean  @default(false)       // نیازمند «خواندم و فهمیدم» (فراتر از read)
  ackDeadline   DateTime?                      // مهلت تایید (برای گزارش پیگیری)
  bannerStyle   Json?                          // {color:"red|amber|blue", icon, dismissible:false}
  attachments   Json?                          // [{name,url,type}] — PDF دستورالعمل و...
  notifyRuleKey String?                        // override قاعده اعلان (اتصال به Gateway)
}

model PostAck {                                 // تاییدخوانی رسمی (الگوی بخشنامه)
  id        String   @id @default(cuid())
  postId    String
  userId    String
  ackAt     DateTime @default(now())
  device    String?
  ip        String?
  location  Json?
  signature String?                             // متن تایید: «خواندم و متعهد می‌شوم»

  @@unique([postId, userId])
  @@index([postId])
}

model SignageScreen {                           // تابلوهای دیجیتال
  id         String  @id @default(cuid())
  name       String                              // «مانیتور دپو شمال»
  location   String?
  pairCode   String  @unique                     // جفت‌سازی صفحه با کد ۶رقمی
  playlistId String?
  isOnline   Boolean @default(false)
  lastSeenAt DateTime?
}

model SignagePlaylist {
  id       String @id @default(cuid())
  name     String
  items    Json                                  // [{type:"post"|"roster_today"|"clock"|"weather"|"custom_html", refId?, seconds}]
  isActive Boolean @default(true)
}
```

## ۴) قابلیت‌های کلیدی

### ۴.۱ هدف‌گیری مخاطب (Audience Builder)
سازنده مخاطب بصری در پنل: نقش‌ها + گروه‌ها + ایستگاه/دپو + **کد شیفت** (فقط شب‌کارهای امشب!) + افراد خاص — با پیش‌نمایش زنده «این اطلاعیه به ۱۴۳ نفر می‌رسد». هدف‌گیری شیفتی از تقویم/لوحه واقعی خوانده می‌شود — قابلیتی که فقط در این پلتفرم یکپارچه ممکن است.

### ۴.۲ چرخه انتشار حرفه‌ای
پیش‌نویس ← بازبینی (نقش تاییدکننده قابل تنظیم per دسته) ← انتشار فوری یا زمان‌بندی‌شده ← انقضای خودکار ← بایگانی. نسخه‌بندی ویرایش‌ها (اطلاعیه منتشرشده ویرایش شود، نسخه قبلی در تاریخچه می‌ماند + نشان «به‌روزرسانی‌شده»).

### ۴.۳ تاییدخوانی و پیگیری (Compliance)
برای `ackRequired`: مودال با اسکرول اجباری تا انتها + دکمه «خواندم و فهمیدم» ← رسید immutable (دستگاه/IP/زمان/موقعیت). داشبورد پیگیری: درصد تایید per واحد، لیست تایید‌نکرده‌ها با دکمه «یادآوری» (پوش) و پس از مهلت، «پیگیری پیامکی» — از طریق Gateway سند قبل. خروجی اکسل رسیدها برای بایگانی رسمی.

### ۴.۴ تابلوی دیجیتال (Signage) — تحول اطلاع‌رسانی فیزیکی
هر مانیتور ایستگاه/دپو با یک مرورگر و **کد جفت‌سازی ۶رقمی** به سیستم وصل می‌شود (صفحه `/signage` تمام‌صفحه، بدون لاگین، فقط‌خواندنی). پلی‌لیست چرخشی: اطلاعیه‌ها + **لوحه امروز** + ساعت/تاریخ جلالی + محتوای HTML سفارشی. به‌روزرسانی زنده با SSE؛ اعلان بحران بلافاصله همه تابلوها را قطع و تمام‌صفحه می‌شود. پایش آنلاین‌بودن تابلوها در پنل.

### ۴.۵ تعامل کنترل‌شده
ری‌اکشن و نظر (موجود) per اطلاعیه قابل روشن/خاموش؛ نظرات با صف تایید اختیاری؛ پرسش‌وپاسخ زیر اطلاعیه‌های مهم (پاسخ رسمی با نشان).

### ۴.۶ اتصال به دروازه اعلان
انتشار ← `notifyEvent("announcement.published", audience, vars)` ← قواعد همان پنل اعلان تعیین می‌کند: فید فقط؟ پوش؟ برای الزامی‌ها پیامکِ دیده‌نشد؟ — دو سیستم، یک زبان.

## ۵) سناریوها

### سناریو ۱ — اطلاعیه الزامی با پیگیری کامل
مدیر ایمنی «دستورالعمل جدید توقف اضطراری» را با PDF پیوست، نوع الزامی، مهلت ۴۸ساعته و مخاطب «همه راهبران» می‌سازد ← تاییدکننده (رئیس بهره‌برداری) در کارتابل تایید می‌کند ← انتشار: مودال اجباری در اولین ورود هر راهبر + پوش ← داشبورد پیگیری: ۸۹٪ تایید در ۲۴ ساعت ← یادآوری پوش به بقیه ← پایان مهلت: ۶ نفر باقی‌مانده پیامک می‌گیرند و لیستشان به سرشیفت‌ها می‌رود ← خروجی اکسل رسیدها بایگانی می‌شود.

### سناریو ۲ — بنر فوری با انقضا
اختلال رادیویی ناحیه ۳ ← اپراتور OCC بنر قرمز «فقط ایستگاه‌های ناحیه ۳» با انقضای ۴ساعته می‌سازد (بدون نیاز به تایید — نوع فوری با permission ویژه) ← نوار قرمز بالای وب/موبایل فقط برای همان مخاطبان ← رفع اختلال: دکمه «پایان» و بنر همه‌جا محو می‌شود.

### سناریو ۳ — تابلوی دپو
نگهبانی دپو یک مانیتور و مینی‌PC دارد ← ادمین «تابلوی جدید» می‌سازد، کد ۶رقمی را در مرورگر مانیتور وارد می‌کند ← پلی‌لیست «دپو»: لوحه امروز (۳۰ث) + اطلاعیه‌های فعال (هرکدام ۱۵ث) + ساعت ← اعلان بحران ساعت ۱۴:۰۲ منتشر می‌شود ← تابلو در همان لحظه قرمز تمام‌صفحه می‌شود.

### سناریو ۴ — هدف‌گیری شیفتی
«جلسه توجیهی فردا ۹ صبح» فقط باید به کسانی برسد که فردا شیفت صبح‌اند ← مدیر در Audience Builder: کد شیفت = صبحِ فردا ← سیستم از لوحه/تقویم، ۴۷ نفر را برمی‌گزیند ← بقیه اسپم نمی‌گیرند.

### سناریو ۵ — انتشار زمان‌بندی‌شده و انقضا
اطلاعیه «قطعی آب دپو، پنجشنبه» یکشنبه نوشته می‌شود، انتشار چهارشنبه ۸ صبح، انقضا جمعه شب — بدون هیچ اقدام دستی دیگری.

## ۶) پنل مدیریت حرفه‌ای و قابل شخصی‌سازی (Admin Console)

```
پنل مدیریت اعلانات
├── 📊 داشبورد              اطلاعیه‌های فعال، نرخ خواندن/تایید، تابلوهای آنلاین، صف بازبینی
├── ✍️ سازنده اطلاعیه        ویرایشگر غنی (متن/تصویر/ویدئو/پیوست) + انتخاب نوع و سطوح نمایش
│                           + Audience Builder با پیش‌نمایش تعداد + زمان‌بندی/انقضا + پیش‌نمایش موبایل/وب/تابلو
├── 🗂️ دسته‌ها و انواع        تعریف دسته‌ها، رنگ/آیکون، نقش تاییدکننده per دسته، مجوز سازنده per نوع
├── ✅ گردش بازبینی           کارتابل تایید/رد با یادداشت؛ قواعد: کدام نوع نیاز به تایید دارد (فوری = بدون تایید با permission)
├── 📈 پیگیری تاییدخوانی      داشبورد per اطلاعیه: درصد، به تفکیک واحد، لیست باقی‌مانده، یادآوری/پیامک، خروجی اکسل
├── 🖥️ تابلوهای دیجیتال       صفحه‌ها + جفت‌سازی، پلی‌لیست‌ساز Drag & Drop، پایش آنلاین، تم تابلو
├── 🎨 قالب‌ها و ظاهر         قالب‌های آماده اطلاعیه، استایل بنرها (رنگ/آیکون/قابلیت بستن)، تم فید
├── 💬 تعامل                 روشن/خاموش نظر و ری‌اکشن per دسته، صف تایید نظرات، پاسخ رسمی
├── 🔔 اتصال اعلان            نگاشت هر نوع/دسته به قاعده Gateway (پوش؟ پیامکِ دیده‌نشد؟)
├── 🧹 چرخه عمر               قواعد بایگانی خودکار، نگهداری رسیدها، پاکسازی منقضی‌ها
└── 🗂️ لاگ حسابرسی            چه کسی چه اطلاعیه‌ای را ساخت/تایید/ویرایش/حذف کرد (before/after)
```

**Permissionها:** `announcements:create` (per نوع)، `announcements:review`، `announcements:urgent` (بنر/بحران بدون تایید)، `announcements:track`، `announcements-admin:manage`، `signage:manage`.

## ۷) API (App Router)

```
── کاربر
GET  /api/announcements?surface=feed|banner|modal      ← فید شخصی‌سازی‌شده بر اساس مخاطب
POST /api/announcements/[id]/read                       ← رسید خواندن
POST /api/announcements/[id]/ack                        ← تایید رسمی (device/ip/location)
GET  /api/announcements/[id]                            ← جزئیات + پیوست‌ها

── تابلو
GET  /api/signage/pair/[code]                           ← جفت‌سازی و دریافت پیکربندی
GET  /api/signage/[screenId]/playlist                   ← محتوا + SSE برای به‌روزرسانی زنده

── ادمین
GET/POST/PATCH/DELETE /api/admin/announcements
POST /api/admin/announcements/[id]/submit|approve|reject|publish|expire
GET  /api/admin/announcements/[id]/tracking             ← آمار تایید + باقی‌مانده‌ها
POST /api/admin/announcements/[id]/remind               ← یادآوری (پوش/پیامک از Gateway)
GET  /api/admin/announcements/[id]/acks/export          ← اکسل رسیدها
GET/POST/PATCH /api/admin/signage/screens  |  /playlists
GET/PATCH /api/admin/announcements/config               ← دسته‌ها، قواعد بازبینی، ظاهر، تعامل
```

## ۹) ملاحظات فنی

- فید با کوئری مخاطب‌آگاه (audience match در DB) + کش کوتاه؛ بنر/مودال در layout سراسری با SWR.
- تابلو: صفحه Next.js سبک، بدون auth با توکن صفحه، خودترمیم اتصال SSE، حالت آفلاین (آخرین پلی‌لیست کش‌شده).
- رسیدهای Ack مانند بخشنامه‌ها immutable و خارج از دسترس ویرایش حتی ادمین.
- اطلاعیه بحران از ساعت سکوت و ترجیحات کاربر عبور می‌کند (هم‌راستا با قواعد Gateway).
- تست: هدف‌گیری مخاطب (شیفتی/ایستگاهی)، گردش بازبینی و انقضا با Vitest.

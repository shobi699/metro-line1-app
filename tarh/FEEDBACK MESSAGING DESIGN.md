# 💬 طرح جامع سامانه بازخورد و پیام سازمانی (Feedback & Messaging Platform)

**نسخه:** 1.0 — سند طراحی و سناریو
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**هدف:** ارتقای ماژول `feedback` فعلی (ثبت + یک پاسخ) به یک **پلتفرم کامل صدای کارکنان**: دسته‌های قابل تعریف با مسیردهی خودکار به نقش مسئول، گفتگوی دوطرفه ردیابی‌پذیر (حتی در حالت ناشناس)، ارجاع و SLA، تابلوی ایده‌ها با رأی، و گزارش‌گیری تحلیلی — با پنل مدیریت کاملاً بدون‌کد.

---

## ۱) زیرساخت موجود که طرح روی آن سوار می‌شود

- مدل `Feedback` (۴ نوع: انتقاد/پیشنهاد/شکایت/قدردانی، پرچم ناشناس، وضعیت ساده، یک پاسخ) — پایه ارتقا.
- الگوهای اثبات‌شده اسناد قبلی: گردش ارجاع (فرم‌ساز)، SLA و Circuit خودکار (فالت)، ناشناسی واقعی (نظرسنجی)، Notification Gateway (اعلان پاسخ)، `AuditLog`، `xlsx`.
- چت بلادرنگ موجود — **مرز:** چت = گفتگوی غیررسمی؛ این سامانه = **پیام رسمی ردیابی‌پذیر** با شماره، وضعیت، مسئول و SLA.

**مسئله فعلی:** بازخورد به «یک صندوق عمومی» می‌رود، معلوم نیست چه کسی مسئول است، گفتگو یک‌باره تمام می‌شود، ناشناس نمی‌تواند ادامه گفتگو دهد، و مدیر هیچ تصویری از روند و گلوگاه‌ها ندارد.

---

## ۲) مفاهیم کلیدی

| مفهوم | توضیح |
| :--- | :--- |
| **دسته (Category)** | تعریف ادمین: «پیشنهاد بهبود»، «شکایت رفاهی»، «گزارش مشکل ایمنی»، «پیام به مدیرعامل»... هر دسته: فرم، مسیر، SLA و سطح محرمانگی خودش |
| **مسیردهی (Routing)** | هر دسته به یک **نقش/تیم مسئول** می‌رسد (نه صندوق عمومی) + قواعد شرطی |
| **گفتگو (Thread)** | پیام‌های دوطرفه روی یک بازخورد — حتی ناشناس (کانال امن بدون افشای هویت) |
| **ارجاع (Referral)** | مسئول می‌تواند به نقش دیگر ارجاع دهد با حفظ تاریخچه |
| **تابلوی ایده‌ها (Idea Board)** | پیشنهادهای عمومی‌شده با رأی همکاران — پیشنهاد پررأی وزن می‌گیرد |

## ۳) مدل داده (Prisma — گسترش سازگار)

```prisma
model FeedbackCategory {
  id            String  @id @default(cuid())
  key           String  @unique               // "safety_issue" | "welfare" | "ceo_message" ...
  title         String
  icon          String?
  description   String?                        // راهنمای ثبت
  assigneeRole  String                         // نقش مسئول پاسخ
  routingRules  Json?                          // شرطی: {if:{field,eq},assignTo:role}
  slaHours      Json                           // {firstResponse: 24, resolve: 120}
  allowAnonymous Boolean @default(true)
  forceAnonymous Boolean @default(false)       // مثل «گزارش تخلف»
  confidential  Boolean @default(false)        // فقط مسئول و بالاتر ببینند (نه گزارش‌های عمومی)
  formFields    Json?                          // فیلدهای اضافی فرم (فرم‌ساز)
  whoCanSubmit  Json?                          // نقش‌های مجاز (خالی = همه)
  ideaBoard     Boolean @default(false)        // پیشنهادها قابل عمومی‌شدن و رأی
  isActive      Boolean @default(true)
  sortOrder     Int     @default(0)
}

// گسترش Feedback موجود:
model Feedback {
  // ...موجود...
  feedbackNo   Int     @unique @default(autoincrement())   // FB-1042
  categoryId   String?
  assigneeId   String?                          // مسئول فعلی
  assigneeRole String?
  priority     String  @default("normal")       // normal | high | critical
  formData     Json?
  attachments  Json?
  anonToken    String? @unique                  // کانال امن ناشناس (پیگیری بدون هویت)
  slaFirstDue  DateTime?
  slaResolveDue DateTime?
  slaBreached  Boolean @default(false)
  satisfaction Int?                             // امتیاز ۱–۵ ثبت‌کننده پس از بستن
  closedAt     DateTime?
  isPublicIdea Boolean @default(false)          // روی تابلوی ایده‌ها
  ideaVotes    Int     @default(0)
}
// وضعیت‌ها گسترش می‌یابد: submitted | in_progress | referred | responded |
//                        resolved | closed | reopened

model FeedbackMessage {                          // گفتگوی دوطرفه
  id         String   @id @default(cuid())
  feedbackId String
  senderKind String                              // submitter | staff
  senderId   String?                             // null اگر ناشناس
  body       String
  attachments Json?
  isInternal Boolean  @default(false)            // یادداشت داخلی تیم (ثبت‌کننده نمی‌بیند)
  createdAt  DateTime @default(now())

  @@index([feedbackId])
}

model FeedbackLog {                              // ارجاع/تغییر وضعیت/SLA — الگوی FaultLog
  id         String   @id @default(cuid())
  feedbackId String
  actorId    String?
  action     String                              // assigned|referred|status|sla_breached|reopened...
  detail     Json?
  createdAt  DateTime @default(now())
  @@index([feedbackId])
}

model IdeaVote {
  feedbackId String
  userId     String
  createdAt  DateTime @default(now())
  @@id([feedbackId, userId])
}
```

> **ناشناسی با گفتگو (نوآوری کلیدی):** ثبت ناشناس ← `anonToken` تصادفی به ثبت‌کننده داده می‌شود (در دستگاهش ذخیره + قابل یادداشت). پیگیری و ادامه گفتگو با توکن، بدون هیچ اتصالی به حساب. پاسخ مسئول ← اعلان بدون‌محتوا («پاسخ جدید برای پیگیری FB-1042») به دستگاه، بدون ثبت هویت سمت سرور.

## ۴) سناریوها

### سناریو ۱ — شکایت رفاهی با گفتگو و رضایت‌سنجی
احمدی «شکایت رفاهی: سرمایش رختکن دپو خراب است» + عکس ثبت می‌کند ← مسیردهی خودکار به نقش «مسئول رفاهی» + SLA پاسخ اول ۲۴س ← مسئول در کارتابل می‌بیند، یادداشت داخلی برای تدارکات می‌گذارد (ثبت‌کننده نمی‌بیند)، به احمدی پاسخ می‌دهد «تا پنجشنبه تعمیر می‌شود» ← احمدی در همان رشته جواب می‌دهد ← پنجشنبه: وضعیت «حل شد» ← احمدی امتیاز ۵ می‌دهد ← در گزارش ماهانه، میانگین رضایت دسته رفاهی بالا می‌رود.

### سناریو ۲ — گزارش ناشناس تخلف با ادامه گفتگو
کارمندی «گزارش تخلف» ثبت می‌کند — دسته `forceAnonymous` و `confidential` ← فقط بازرس ارشد می‌بیند ← توکن پیگیری به ثبت‌کننده ← بازرس سوال تکمیلی می‌پرسد ← ثبت‌کننده با توکن، بدون افشای هویت پاسخ می‌دهد ← رسیدگی کامل با ناشناسی حفظ‌شده. (کاری که سیستم فعلی اصلاً نمی‌تواند.)

### سناریو ۳ — ارجاع بین‌واحدی با SLA
«پیشنهاد بهبود» درباره چیدمان لوحه به مسئول بهره‌برداری می‌رسد ← موضوع فنی است، **ارجاع** به «مسئول سامانه» با یادداشت ← ساعت SLA از نو برای واحد جدید ← تاریخچه کامل ارجاع در FeedbackLog ← اگر ۱۲۰ ساعت بی‌پاسخ بماند: نقض SLA و تشدید به مدیر.

### سناریو ۴ — تابلوی ایده‌ها
پیشنهاد «نصب آبسردکن سکوی تجریش» با اجازه ثبت‌کننده عمومی می‌شود ← همکاران رأی می‌دهند ← ۱۴۷ رأی ← در داشبورد مدیر بالای فهرست «ایده‌های پرتقاضا» ← تصویب و اجرا ← وضعیت «اجرا شد» با نشان 🏆 و اعلان به همه رأی‌دهندگان — چرخه «صدایت شنیده شد» کامل می‌شود.

### سناریو ۵ — پیام مستقیم به مدیرعامل
دسته «پیام به مدیرعامل»: فقط دفتر مدیرعامل می‌بیند، SLA ۷۲س، ناشناسی مجاز ← کانال رسمی و ردیابی‌پذیر که جای شایعه و واسطه را می‌گیرد.

## ۵) گزارش‌گیری و تحلیل

- **داشبورد مدیریتی:** حجم per دسته/واحد/ماه، میانگین زمان پاسخ اول و حل، نرخ نقض SLA، میانگین رضایت، نرخ بازگشایی، نقشه گرمایی موضوعات per ایستگاه/دپو.
- **گزارش گلوگاه:** کدام نقش/واحد کندترین پاسخ را دارد؛ کدام دسته بیشترین شکایت تکرارشونده.
- **روند صدای کارکنان:** نسبت انتقاد/پیشنهاد/قدردانی در زمان — دماسنج سازمان.
- **کارنامه ایده‌ها:** ثبت‌شده ← عمومی ← پررأی ← اجراشده (قیف نوآوری).
- خروجی **اکسل** با فیلتر (در دسته‌های ناشناس بدون هویت) + PDF مدیریتی + چاپ. (فاز بعد: خوشه‌بندی موضوعی متن‌ها با لایه AI موجود.)

## ۶) پنل مدیریت حرفه‌ای و قابل شخصی‌سازی (Admin Console)

```
پنل مدیریت بازخورد
├── 📊 داشبورد            حجم، SLA، رضایت، گلوگاه‌ها، ایده‌های پرتقاضا
├── 🗂️ دسته‌ها             ساخت دسته: عنوان/آیکون/راهنما + نقش مسئول + قواعد مسیردهی شرطی
│                          + SLA دومرحله‌ای + ناشناسی (مجاز/اجباری) + محرمانگی + فرم اختصاصی
│                          + چه نقشی ثبت کند + تابلوی ایده — همه بدون کد
├── 👥 مسئولان و ظرفیت      نگاشت نقش←کاربران پاسخگو، توزیع خودکار (round-robin)، جانشین در مرخصی
├── ⏱️ SLA و تشدید          پیش‌فرض‌ها + پله‌های تشدید (یادآور مسئول ← اطلاع مدیر) از Gateway
├── ✉️ پاسخ‌های آماده        Canned Responses per دسته با متغیر — سرعت و یکدستی پاسخ
├── 💡 تابلوی ایده‌ها        قواعد عمومی‌شدن (تایید ثبت‌کننده + بازبین)، آستانه رأی، اعلان اجرا
├── 🔒 حریم خصوصی           سیاست ناشناسی (غیرقابل برگشت)، نگهداری داده، دسترسی محرمانه‌ها
├── 📈 مرکز گزارش            همه گزارش‌های بخش ۵ + زمان‌بندی گزارش ماهانه خودکار به مدیران
├── 🔔 اعلان‌ها              نگاشت رویدادها (ثبت/پاسخ/نقض/اجرا) به قواعد Gateway
└── 🗂️ لاگ حسابرسی           هر تغییر پیکربندی و هر اقدام روی بازخورد
```

**Permissionها:** `feedback:submit`، `feedback:respond` (per دسته از طریق نقش مسئول)، `feedback:refer`، `feedback:view-confidential`، `feedback:reports`، `feedback-admin:manage`.

## ۷) API (App Router)

```
── کاربر
GET  /api/feedback/categories                  ← دسته‌های مجاز من + فرم هر یک
POST /api/feedback                             ← ثبت (اگر ناشناس: برگشت anonToken)
GET  /api/feedback/my                          ← بازخوردهای من + وضعیت
GET  /api/feedback/track/[anonToken]           ← پیگیری ناشناس
POST /api/feedback/[id|token]/messages         ← ادامه گفتگو
POST /api/feedback/[id]/satisfaction {1..5}
GET  /api/feedback/ideas?sort=votes            ← تابلوی ایده‌ها
POST /api/feedback/ideas/[id]/vote

── مسئول
GET  /api/feedback/inbox?status=&category=     ← کارتابل من (نقش‌محور)
POST /api/feedback/[id]/assign|refer|status
POST /api/feedback/[id]/messages {isInternal?}
GET  /api/feedback/canned?category=

── ادمین
GET/POST/PATCH /api/admin/feedback/categories
GET/PATCH      /api/admin/feedback/config      ← SLA/تشدید/حریم/ایده‌ها
GET/POST       /api/admin/feedback/canned
GET            /api/admin/feedback/reports?type=...  |  /export
```

## ۸) UI (وب + موبایل)

- **ثبت (موبایل):** انتخاب دسته کارت‌دار با راهنما ← فرم کوتاه دسته + عکس از دوربین ← سوییچ ناشناس با توضیح شفاف ← شماره پیگیری بزرگ + (ناشناس: نمایش توکن با «کپی/ذخیره»).
- **پیگیری:** رشته گفتگوی چت‌مانند + نوار وضعیت مرحله‌ای (ثبت ← بررسی ← پاسخ ← حل).
- **کارتابل مسئول (وب):** صف با نشان SLA رنگی، پاسخ آماده، یادداشت داخلی با پس‌زمینه متمایز، اقدام گروهی.
- **تابلوی ایده‌ها:** کارت‌های رأی‌دار با مرتب‌سازی، نشان «اجرا شد».
- RTL، اعداد فارسی، دارک‌مود، حالات کامل.

## ۱۰) ملاحظات فنی

- ناشناسی در لایه ذخیره (بدون userId/IP در دسته‌های ناشناس)؛ anonToken با entropy کافی و قابل ابطال توسط خود ثبت‌کننده.
- محرمانه‌ها (`confidential`) در همه گزارش‌ها و جستجوها فیلتر سرور‌ی می‌شوند.
- ضدسوءاستفاده: نرخ‌محدودی ثبت، و امکان «بستن با برچسب نامرتبط» بدون حذف (Audit کامل).
- تست: مسیردهی شرطی، SLA دومرحله‌ای، جریان توکن ناشناس، رأی یکتا با Vitest.

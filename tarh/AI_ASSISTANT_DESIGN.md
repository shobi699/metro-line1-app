# 🤖 طرح جامع دستیار هوش مصنوعی نسل بعد (AI Assistant Platform)

**نسخه:** 1.0 — سند طراحی و سناریو
**پروژه:** metro-line1-app — سامانه سیر و حرکت خط ۱ مترو تهران
**هدف:** ارتقای دستیار AI فعلی به یک **پلتفرم دستیار سازمانی نقش‌آگاه** با دو اولویت مطلق: **کمترین هزینه** و **بالاترین سرعت پاسخ** — از طریق معماری پاسخ‌دهی چندلایه (بیشترِ پاسخ‌ها اصلاً به LLM نرسند)، دستیارهای اختصاصی per نقش، ابزارمندی روی داده‌های سیستم، و پنل مدیریتی که مدل‌ها، دانش، دسترسی و بودجه را بدون کدنویسی کنترل می‌کند.

---

## ۱) زیرساخت موجود (نقطه قوت واقعی پروژه)

- **AIGateway** با مسیریابی اولویت‌دار بین تأمین‌کننده‌ها (`AiProvider` در DB)، Circuit Breaker (۳ خطا ← ۵ دقیقه قطع)، timeout و fallback — آماده و تست‌شده (`gateway.test.ts`).
- آداپتورها: Gemini + OpenAI-compatible (پوشش OpenRouter، **Ollama محلی**، و هر سرویس سازگار — از جمله ارائه‌دهندگان داخلی ایران).
- **Embedding محلی** (`@xenova/transformers`، ۳۸۴بعدی، `cosineSimilarity`) + **کش معنایی** (`AiKnowledgeCache`) با هدف <۵۰ms — پایه‌ی «هزینه صفر» موجود است.
- RAG روی آیین‌نامه‌ها، `KnowledgeArticle/KnowledgeFAQ`، پنل‌های `ai-providers` و `ai-cache`.

**فرصت:** این اجزا الان «یک چت عمومی» را پشتیبانی می‌کنند. طرح، آن‌ها را به سیستم پاسخ‌دهی هوشمندِ لایه‌بندی‌شده و نقش‌آگاه ارتقا می‌دهد.

## ۲) معماری پاسخ چندلایه (هزینه↓ سرعت↑)

**اصل طلایی: گران‌ترین لایه، آخرین لایه.** هر پرسش از نردبان زیر بالا می‌رود و در اولین لایه‌ی قانع‌کننده متوقف می‌شود:

```
پرسش کاربر
  │
  ▼ L0. پاسخ ساخت‌یافته (۰ توکن، <10ms)
  │   تشخیص intent با کلیدواژه/الگو: «شیفت فردام چیه؟» «قطار ۱۱۰ چه فالتی داره؟»
  │   ← پاسخ مستقیم از API ماژول‌ها (تقویم/لوحه/فالت/فرم) — بدون هیچ LLM
  ▼ L1. کش معنایی (۰ توکن، <50ms) — موجود، ارتقایافته
  │   شباهت ≥ آستانه (پیش‌فرض 0.92) با پاسخ کش‌شده‌ی هم‌نقش ← پاسخ فوری
  │   + کش «per نقش» (پاسخ راهبر ≠ پاسخ مدیر) + TTL per دسته دانش
  ▼ L2. FAQ و دانش تأییدشده (۰ توکن، <80ms)
  │   تطبیق معنایی با KnowledgeFAQ — پاسخ رسمیِ نوشته‌ی انسان + استناد
  ▼ L3. RAG با مدل کوچک/محلی (کم‌هزینه، سریع)
  │   بازیابی top-k از دانش نقش‌مجاز ← LLM «رده اقتصادی» (Ollama محلی یا مدل ارزان)
  │   ← پاسخ با استناد ← در کش معنایی ذخیره می‌شود (سوال بعدیِ مشابه = L1)
  ▼ L4. مدل قوی (فقط پیچیده‌ها — درصد کم)
      قواعد ارتقا: طول/پیچیدگی، درخواست تحلیل، شکست L3 — از Gateway موجود
```

**تخمین اثر:** در بار واقعی سازمانی (سوالات تکراری آیین‌نامه/شیفت)، انتظار می‌رود ۷۰–۸۵٪ پرسش‌ها در L0–L2 (هزینه صفر، زیر ۱۰۰ms) پاسخ بگیرند؛ عدد دقیق در داشبورد همین طرح اندازه‌گیری و بهینه می‌شود.

**سرعت ادراکی:** پاسخ‌های L3/L4 با **استریم توکنی (SSE)** نمایش داده می‌شوند — کاربر از کلمه اول می‌خواند.

## ۳) دستیار نقش‌آگاه (پاسخ به «برای هر نقش»)

`AiPersona` قابل تعریف در پنل — هر نقش، دستیارِ خودش:

| پرسونا | مخاطب | دانش مجاز | ابزارها | مدل پیش‌فرض |
| :--- | :--- | :--- | :--- | :--- |
| **دستیار راهبر** | operator | آیین‌نامه‌ها، ایمنی، FAQ عمومی | شیفت من، لوحه امروز، ثبت فالت | اقتصادی |
| **دستیار سرشیفت** | supervisor | + دستورالعمل‌های مدیریتی | + کارتابل‌ها، آمار ناحیه | اقتصادی/قوی |
| **دستیار تعمیرات** | maintenance | + کاتالوگ خطا و سوابق فالت | تاریخچه قطار، فالت‌های باز | اقتصادی |
| **دستیار مدیر** | manager | + گزارش‌ها و KPI | کوئری تحلیلی گزارش‌ها | قوی |

- **جداسازی دانش در بازیابی:** فیلتر مجوز نقش قبل از RAG اعمال می‌شود — دانش محرمانه حتی وارد پرامپت نمی‌شود (نه اینکه فقط نمایش داده نشود).
- **ابزارها (Tool Calls) روی APIهای موجود:** «شیفت پنجشنبه‌ام چیه؟» ← فراخوانی تقویم؛ «برای قطار ۱۱۰ فالت ترمز ثبت کن» ← پیش‌نویس فرم فالت با تایید کاربر. هر ابزار per پرسونا روشن/خاموش و **همیشه در چارچوب RBAC خود کاربر** اجرا می‌شود.

## ۴) سناریوها

### سناریو ۱ — سوال تکراری آیین‌نامه (مسیر هزینه‌صفر)
راهبر: «حداکثر سرعت در TSR چقدره؟» ← L1: مشابهت 0.96 با کشِ نقش راهبر ← پاسخ در ۴۰ms با استناد، هزینه صفر. در داشبورد: hit کش +۱.

### سناریو ۲ — سوال داده‌ای زنده (L0)
«فردا شیفتم چیه و با کدوم قطار؟» ← intent «شیفت من» ← API تقویم/لوحه ← «فردا صبح ۰۶:۰۰–۱۴:۰۰، قطار ۱۱۸، اعزام ۰۶:۱۵ تجریش» — بدون LLM، <10ms، همیشه دقیق (نه توهم‌پذیر).

### سناریو ۳ — سوال جدید و ذخیره در کش
«در برف شدید، فاصله‌گیری ترمز چطور تغییر می‌کند؟» ← L1/L2 miss ← L3: بازیابی ۳ بند آیین‌نامه ← مدل اقتصادی، پاسخ استریمی با استناد ← ذخیره در کش نقش راهبر ← ۲۰ راهبر بعدی که مشابهش را می‌پرسند: L1.

### سناریو ۴ — اقدام با ابزار
تعمیرکار: «فالت‌های باز قطار ۱۲۵ رو بده و برای HVAC واگن ۵ شروع تعمیر بزن» ← ابزار فالت: لیست ← کارت تایید اقدام ← لمس تایید ← ثبت واقعی از API با هویت خودش + لاگ.

### سناریو ۵ — کنترل بودجه
نیمه ماه، مصرف به ۸۰٪ سقف می‌رسد ← هشدار به ادمین ← قاعده خودکار (قابل تنظیم): L4 موقتاً فقط برای مدیران؛ بقیه تا L3 ← پایان ماه بدون عبور از بودجه، بدون قطع سرویس.

### سناریو ۶ — بهبود مستمر از بازخورد
پاسخ‌های 👎 در صف بازبینی ← کارشناس دانش، پاسخ درست را به FAQ اضافه می‌کند ← از این پس همان سوال در L2 (رسمی و رایگان) پاسخ می‌گیرد — سیستم هر هفته ارزان‌تر و دقیق‌تر می‌شود.

## ۵) گزارش‌گیری و تحلیل (پاسخ به «قابل گزارش‌گیری»)

- **داشبورد هزینه/سرعت:** توزیع پاسخ‌ها per لایه (نرخ hit هر لایه)، میانگین تأخیر per لایه، توکن و هزینه per تأمین‌کننده/مدل/روز، **«صرفه‌جویی کش»** به ریال/توکن (سوالاتی که LLM نرفتند × هزینه واحد).
- **کیفیت:** نرخ 👍/👎 per پرسونا و دسته دانش، پرتکرارترین سوالات بی‌پاسخ (ورودی تولید FAQ)، پاسخ‌های بدون استناد.
- **مصرف per نقش/واحد:** چه کسی چقدر می‌پرسد؛ سقف‌های نزدیک به پر شدن.
- **سلامت:** وضعیت Circuit هر تأمین‌کننده، خطاها، تأخیر p95.
- خروجی **اکسل** همه گزارش‌ها + گزارش ماهانه خودکار به مدیر (از Gateway اعلان).

## ۶) پنل مدیریت حرفه‌ای و قابل شخصی‌سازی (Admin Console)

```
پنل مدیریت دستیار AI
├── 📊 داشبورد            hit لایه‌ها، هزینه امروز/ماه، صرفه‌جویی کش، تأخیر p50/p95، رضایت
├── 🧠 تأمین‌کننده‌ها و مدل‌ها   (ارتقای ai-providers موجود) افزودن provider با کلید، اولویت
│                          Drag & Drop، قیمت per 1K توکن (برای گزارش هزینه)، تست اتصال،
│                          نگاشت «رده اقتصادی/قوی» ← مدل مشخص — تغییر مدل = یک انتخاب
├── 🪜 قواعد لایه‌بندی        آستانه شباهت کش (per دسته)، TTL کش، قواعد ارتقا به L4،
│                          روشن/خاموشی هر لایه — تیونینگ هزینه/کیفیت بدون دیپلوی
├── 👥 پرسوناها              ساخت دستیار per نقش: نام/آیکون/لحن (system prompt ویرایش‌پذیر)،
│                          دانش مجاز (انتخاب دسته‌ها)، ابزارهای فعال، مدل پیش‌فرض، سقف مصرف
├── 📚 مدیریت دانش           منابع RAG: آپلود سند ← chunking و embedding خودکار ← دسته و سطح دسترسی؛
│                          FAQ رسمی (L2) با ویرایشگر؛ بازindex یک‌کلیکه؛ نسخه‌بندی اسناد
├── 🧰 ابزارها               فهرست ابزارهای سیستمی (شیفت/لوحه/فالت/فرم/گزارش) + مجوز per پرسونا
│                          + «نیازمند تایید کاربر» per ابزار
├── 💰 بودجه و سهمیه          سقف ماهانه کل/تأمین‌کننده/نقش/کاربر + قواعد رفتار در آستانه (سناریو ۵)
├── 🗨️ بازبینی گفتگوها        نمونه‌گیری، جستجو، صف 👎 ← تبدیل به FAQ؛ سیاست نگهداری/حذف
├── 🛡️ حريم و ایمنی            ماسک‌کردن داده حساس در پرامپت، فهرست ممنوعه، فقط‌داخلی (Ollama) برای
│                          دسته‌های محرمانه، بنر شفافیت «پاسخ AI است»
├── 📈 مرکز گزارش             همه گزارش‌های بخش ۵ + اکسل + زمان‌بندی ارسال
└── 🗂️ لاگ حسابرسی            هر تغییر پیکربندی با before/after
```

**Permissionها:** `ai:chat` (per پرسونا از طریق نقش)، `ai:tools` ، `ai:knowledge-manage`، `ai:review`، `ai:reports`، `ai-admin:manage`.

## ۷) مدل داده (Prisma — افزایشی)

```prisma
model AiPersona {
  id           String  @id @default(cuid())
  key          String  @unique              // "operator" | "supervisor" | ...
  title        String
  icon         String?
  systemPrompt String
  roleKeys     Json                          // نقش‌های مجاز
  knowledgeCats Json                         // دسته‌های دانش مجاز
  tools        Json                          // ابزارهای فعال + نیاز به تایید
  economyModel String?                       // نگاشت رده‌ها
  strongModel  String?
  monthlyTokenCap Int?
  isActive     Boolean @default(true)
}

model AiKnowledgeSource {                     // اسناد RAG
  id         String   @id @default(cuid())
  title      String
  category   String
  accessRoles Json?
  fileUrl    String?
  version    Int      @default(1)
  chunkCount Int      @default(0)
  indexedAt  DateTime?
  isActive   Boolean  @default(true)
}
// + AiChunk(embedding Bytes, sourceId, text, meta) برای بازیابی

model AiInteraction {                         // پایه گزارش‌گیری
  id          String   @id @default(cuid())
  userId      String
  personaKey  String
  layer       String                          // L0|L1|L2|L3|L4
  provider    String?
  model       String?
  tokensIn    Int      @default(0)
  tokensOut   Int      @default(0)
  costEst     Float    @default(0)
  latencyMs   Int
  rating      Int?                            // 1 / -1
  toolUsed    String?
  createdAt   DateTime @default(now())

  @@index([createdAt])
  @@index([personaKey, layer])
}
// AiKnowledgeCache موجود گسترش: personaKey, ttlAt, hitCount, sourceRefs
```

## ۱۰) ملاحظات فنی

- **هزینه از طراحی، نه صرفه‌جویی بعدی:** کش و FAQ قبل از LLM؛ context کوتاه (فقط chunkهای بازیابی‌شده، نه کل سند)؛ خلاصه‌سازی تاریخچه گفتگو بعد از N پیام؛ max_tokens per پرسونا.
- **Ollama محلی = هم هزینه صفر هم مصونیت تحریم** برای L3؛ ارائه‌دهندگان خارجی فقط لایه آخر و قابل خاموشی کامل از پنل (هم‌راستا با سند اعلان: استقلال زیرساختی).
- توهم‌زدایی: L0 برای داده‌های واقعی (شیفت/فالت) اجباری است — LLM هرگز منبع «داده زنده» نیست؛ پاسخ‌های L3/L4 همیشه با استناد و در صورت نبود منبع: «در آیین‌نامه‌ها نیافتم» (قابل تنظیم در پرسونا).
- embedding و chunking در background job؛ ایندکس برداری در LibSQL (جدول AiChunk) با پیش‌فیلتر دسته/نقش سپس cosine — برای مقیاس فعلی کافی، مسیر ارتقا به pgvector/Qdrant باز.
- تست: مسیر لایه‌ها (hit/miss)، فیلتر دانش نقش، قواعد بودجه، ابزار با تایید — Vitest، الگوی تست‌های موجود gateway/embedding.
